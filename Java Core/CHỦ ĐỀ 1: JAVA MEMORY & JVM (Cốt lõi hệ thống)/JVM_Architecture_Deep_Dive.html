Java Core JVM Internals Memory Model

1. JVM là gì? Tại sao phải học JVM?

Java Virtual Machine (JVM) là một **máy ảo** đóng vai trò là môi trường thực thi cho các ứng dụng Java. Nó chuyển đổi mã Bytecode thành mã máy (Native Machine Code) mà CPU có thể hiểu được.

Triết lý thiết kế

  • Platform Independence: Bytecode giống nhau trên Windows/Linux, chỉ cần JVM tương ứng.
  • Automatic Memory Management: Giải phóng lập trình viên khỏi nỗi lo malloc/free nhờ Garbage Collector.

Phân biệt JDK, JRE, JVM

  • JVM: Trái tim thực thi code.
  • JRE: JVM + Libraries (Để chạy app).
  • JDK: JRE + Tools (javac, debugger... Để phát triển app).

3. Class Loader Subsystem: "Cánh cổng vào"

JVM không load tất cả class ngay khi khởi động. Nó load theo kiểu **Lazy Loading** (cần mới load).

1

Loading (Cơ chế ủy quyền - Delegation Model)

Khi một class được yêu cầu, ClassLoader sẽ hỏi cha của nó trước (theo thứ tự: Bootstrap -> Extension -> Application). Nếu cha không tìm thấy, con mới tự load. Điều này giúp bảo mật, tránh việc người dùng tạo class giả mạo (vd: java.lang.String).

2

Linking

Verify (Kiểm tra bytecode hợp lệ) -> Prepare (Cấp phát memory cho static variables với giá trị default) -> Resolve (Thay thế các tham chiếu ký hiệu bằng địa chỉ thực).

3

Initialization

Đây là lúc các khối static { ... } được thực thi và các biến static được gán giá trị thực sự.

4. Runtime Data Areas: Bộ não và Bộ nhớ

Stack Area (Per Thread)

Mỗi Thread khi sinh ra sẽ có một Stack riêng. Nó lưu trữ các **Stack Frames**.

  • Lưu biến cục bộ (primitive types, object references).
  • Hoạt động theo cơ chế LIFO.
  • Lỗi: StackOverflowError (do đệ quy vô tận).

Heap Area (Shared)

Dùng chung cho toàn bộ ứng dụng. Là nơi "cư ngụ" của tất cả các **Objects** và **Arrays**.

  • Được quản lý bởi Garbage Collector.
  • Chia thành: Young Generation, Old Generation.
  • Lỗi: OutOfMemoryError (khi heap đầy).

5. Execution Engine: Cỗ máy tốc độ

Tại sao Java "chậm" lúc đầu nhưng "nhanh" về sau? Câu trả lời là **JIT (Just-In-Time) Compiler**.

Interpreter

Đọc Bytecode và thực thi từng dòng. Nhanh lúc bắt đầu nhưng chậm nếu lặp lại nhiều lần.

JIT Compiler

Theo dõi các "Hot Spots" (đoạn code chạy nhiều). Nó compile thẳng đoạn code đó sang mã máy và lưu vào cache. Lần sau gọi lại sẽ cực nhanh!

6. Tư duy Senior: Performance & Optimization

Hiểu về -Xms và -Xmx

Trong Production, hãy luôn đặt -Xms (Initial Heap) bằng -Xmx (Max Heap). Việc này giúp JVM không phải tốn tài nguyên để co giãn vùng nhớ Heap liên tục, từ đó giảm Pause Time của Garbage Collector.

Cảnh giác với Memory Leak

Dù có Garbage Collector, nhưng nếu bạn giữ tham chiếu đến một đối tượng không còn dùng (ví dụ: static list chứa mọi request), GC sẽ không thể dọn dẹp nó. Đây chính là nguyên nhân gây chết app âm thầm.

Kiểm tra kiến thức (Interview Style)

Câu 1: Chuyện gì xảy ra nếu bạn gọi một hàm đệ quy không có điểm dừng?

JVM sẽ sinh ra StackOverflowError. Vì mỗi lần gọi hàm, một Frame mới được đẩy vào Stack. Khi Stack đầy (vượt quá kích thước quy định bởi -Xss), lỗi sẽ xảy ra.

Câu 2: Tại sao chúng ta không thể load lại class java.lang.String bằng ClassLoader tự tạo?

Nhờ cơ chế Parent Delegation. ClassLoader của bạn sẽ hỏi cha nó trước. Bootstrap ClassLoader (cao nhất) đã load java.lang.String chuẩn của JDK rồi, nên nó sẽ trả về class đó, ngăn chặn việc nạp class giả mạo.

Câu 3: Metaspace (Java 8+) khác gì so với PermGen (Java 7-)?

PermGen có kích thước cố định, dễ gây java.lang.OutOfMemoryError: PermGen space. Metaspace sử dụng bộ nhớ Native (RAM của OS), nó có thể tự mở rộng linh hoạt, giúp hạn chế lỗi OOM cho metadata của class.
