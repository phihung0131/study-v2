Bài 1.2: Stack vs Heap

"Stack là ngắn hạn, Heap là dài hạn. Hiểu sự khác biệt để làm chủ hiệu năng."

1. Phép ẩn dụ: Balo và Kho bãi

Stack (Chiếc Balo)

Hãy tưởng tượng em đi leo núi. Những thứ em cần dùng ngay (biến cục bộ, địa chỉ hàm) em bỏ vào **Balo**. Nó nhỏ, nhẹ, lấy ra cất vào cực nhanh. Nhưng leo xong đoạn dốc đó (kết thúc hàm), em sẽ đổ sạch đồ trong Balo ra để đi đoạn tiếp theo.

Heap (Cái Kho)

Những vật dụng cồng kềnh (Objects) em gửi ở **Kho bãi**. Nó rộng thênh thang, chứa được cả tấn đồ. Để lấy đồ, em phải cầm một **tấm thẻ (Reference)** ghi số phòng. Khi em không dùng nữa, chủ kho (Garbage Collector) sẽ đi kiểm tra và dọn dẹp sau.

2. Phân tích chuyên sâu: Stack Memory

Cơ chế hoạt động

  • LIFO (Last-In-First-Out): Hàm nào gọi sau cùng sẽ thoát ra trước cùng.
  • Stack Frame: Mỗi khi gọi một method, một Frame được đẩy vào. Chứa:
    • Local Variable Array (Biến cục bộ).
    • Operand Stack (Dùng cho tính toán toán học).
    • Frame Data (Tham chiếu tới Runtime Constant Pool).
  • Tính chất: Được quản lý bởi hệ điều hành/JVM, không cần GC dọn dẹp.

3. Phân tích chuyên sâu: Heap Memory

Cơ chế hoạt động

  • Vùng nhớ dùng chung: Mọi Thread đều có thể truy cập vào đối tượng trên Heap (nếu có tham chiếu).
  • Phân mảnh bộ nhớ: Do Objects có kích thước khác nhau và "sống" lâu mau khác nhau, Heap dễ bị phân mảnh.
  • Garbage Collection: Là đối tượng chính của các thuật toán GC.

JVM Internals: TLAB (Thread Local Allocation Buffer)
Để tránh các Thread tranh giành nhau khi tạo Object trên Heap, JVM cấp cho mỗi Thread một vùng nhỏ trên Heap gọi là TLAB để tự ý "vẽ" Object vào đó mà không cần Lock. Cực kỳ quan trọng cho Performance!

4. Code thực chiến: Chuyện gì đang xảy ra?

// Tình huống: Pass-by-value trong Java liên quan gì đến Stack/Heap?
public class MemoryTest {
    public static void main(String[] args) {
        int age = 25; // Stack: Lưu giá trị 25
        User user = new User("Tuan"); // Stack: Lưu địa chỉ (0x123). Heap: Lưu data User.
        
        update(age, user);
        
        System.out.println(age);  // Vẫn là 25 (vì Stack frame cũ không đổi)
        System.out.println(user.getName()); // Đổi thành "Tèo" (vì cùng trỏ vào 1 vùng Heap)
    }

    static void update(int a, User u) {
        a = 30; // Chỉ thay đổi bản copy trong Stack Frame của hàm update
        u.setName("Tèo"); // Thay đổi data thực sự trong "Cái Kho" (Heap)
    }
}

Tuyệt chiêu tối ưu: Escape Analysis

"Thưa Thầy, có bao giờ Object được lưu trên Stack không?" -> **CÓ**. Nếu JVM nhận ra một Object chỉ được sử dụng bên trong một hàm và không bao giờ "thoát" (escape) ra ngoài (ví dụ: trả về hoặc gán cho static), JVM sẽ tự động cấp phát Object đó ngay trên **Stack**.

Lợi ích: Không tốn công dọn rác (GC), tốc độ thực thi tăng vọt. Đây là một trong những tối ưu hóa đỉnh cao của JIT Compiler.

Câu hỏi phỏng vấn (Hàng nóng)

Q1: Tại sao việc sử dụng biến toàn cục (Static) quá nhiều lại dễ gây OutOfMemory?

A: Vì các biến Static được tham chiếu từ "GC Roots", chúng sẽ tồn tại vĩnh viễn trong Heap cho đến khi app tắt. GC sẽ không bao giờ dám dọn dẹp các đối tượng này.

Q2: Tăng kích thước Stack (-Xss) có luôn tốt không?

A: Không. Tổng bộ nhớ RAM là hữu hạn. Nếu mỗi Stack của 1 Thread quá lớn, bạn sẽ tạo được ít Thread hơn. Trade-off ở đây là: Đệ quy sâu hơn vs. Khả năng xử lý đa luồng (Concurrency).

Q3: Chuyện gì xảy ra nếu ta gán user = null?

A: Biến tham chiếu trên **Stack** sẽ mang giá trị null. Đối tượng thực sự trên **Heap** trở thành "unreachable" (mồ côi). Lúc này nó chính thức trở thành "rác" để GC dọn dẹp.
