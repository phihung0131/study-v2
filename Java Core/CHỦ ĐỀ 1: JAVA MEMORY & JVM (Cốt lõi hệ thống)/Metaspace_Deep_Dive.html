JVM INTERNALS

Bài 1.3: Metaspace (Java 8+)

Tại sao PermGen biến mất? Và cách Java quản lý Metadata ở vùng nhớ Native.

1. Cuộc cách mạng: Từ PermGen đến Metaspace

Kẻ tiền nhiệm: PermGen

Nằm bên trong JVM Heap. Có kích thước cố định (mặc định rất nhỏ, khoảng 64MB-82MB). Khi nạp nhiều class (thời đại web app bùng nổ), nó thường xuyên "vỡ trận" gây lỗi java.lang.OutOfMemoryError: PermGen space.

Kẻ kế nhiệm: Metaspace

Nằm ở Native Memory (RAM của hệ điều hành, không thuộc Heap). Mặc định, Metaspace có thể tự động co giãn theo nhu cầu của ứng dụng, bị giới hạn bởi chính dung lượng RAM của máy chủ.

Mô hình bộ nhớ Java 8+

Total OS Process Memory
Java Heap Area (Young + Old)
Native Memory (Off-Heap)
Metaspace
Thread Stacks / Code Cache

* Lưu ý: Metaspace không còn bị giới hạn bởi kích thước Heap (-Xmx).

2. Bên trong Metaspace có gì?

Metaspace lưu trữ tất cả thông tin mà JVM cần để biết "Class này là ai?".

Class Metadata

Tên class, cấu trúc phân cấp (cha/con), các interface thực thi.

Method Data

Tên hàm, kiểu dữ liệu trả về, danh sách tham số, Bytecode của hàm.

Constant Pool

Các hằng số liên quan đến class (chứ không phải String Pool nhé!).

Lưu ý cực quan trọng của một Chuyên gia:

Từ Java 7, Static Variables (biến tĩnh) và String Pool đã được chuyển từ PermGen sang Java Heap. Metaspace chỉ lưu Metadata (mô tả về cấu trúc lớp).

3. Cấu hình & Tối ưu (Tuning)

Dù Metaspace có thể tự co giãn, nhưng trong Production, "thả rông" là một thảm họa. Nếu một ClassLoader bị leak, nó sẽ nuốt sạch RAM của server.

# Cấu hình khuyến nghị cho JVM
-XX:MetaspaceSize=256m // Ngưỡng bắt đầu dọn rác (GC) cho Metaspace
-XX:MaxMetaspaceSize=512m // Giới hạn cứng để tránh "nuốt" sạch RAM OS
Hệ quả: Nếu bạn đạt đến MaxMetaspaceSize, JVM sẽ văng lỗi OutOfMemoryError: Metaspace. Lúc này, hãy kiểm tra xem bạn có đang dùng các thư viện sinh bytecode động (như CGLIB, Javassist) quá đà không.

4. Case Study: ClassLoader Leak

"Thầy ơi, tại sao app của em chạy 3 ngày là sập, dù Heap vẫn còn trống?"

Đây là kịch bản phổ biến trong các ứng dụng sử dụng **Hot Deployment** (như Tomcat reload context). Mỗi lần reload, một ClassLoader mới được sinh ra để nạp lại các class.

  • ClassLoader cũ lẽ ra phải bị GC dọn đi.
  • Nhưng nếu một Thread bên ngoài (ví dụ một con Thread tĩnh) vẫn giữ tham chiếu tới một Class thuộc ClassLoader cũ.
  • ClassLoader cũ KHÔNG bị dọn -> Toàn bộ metadata của nó trong Metaspace KHÔNG được giải phóng.
  • Qua nhiều lần reload, Metaspace tràn -> Server die.

Câu hỏi phỏng vấn (Deep Level)

Q1: Tại sao việc chuyển sang Metaspace lại giúp tăng tốc độ Garbage Collection?
A: Trước đây, PermGen là một phần của Heap, GC phải quét qua nó mỗi khi dọn dẹp bộ nhớ. Metaspace nằm ở Native Memory, việc cấp phát và dọn dẹp nó được tách biệt hoàn toàn. Hơn nữa, Metaspace sử dụng các "chế độ dọn dẹp" đơn giản hơn, giảm tải cho Full GC.
Q2: Compressed Class Space là gì?
A: Trên hệ điều hành 64-bit, con trỏ đối tượng có thể rất lớn. JVM sử dụng một kỹ thuật gọi là "Compressed Oops" và dành riêng một vùng trong Metaspace gọi là **Compressed Class Space** để lưu trữ các tham chiếu lớp một cách gọn nhẹ hơn, nhằm tiết kiệm bộ nhớ.
