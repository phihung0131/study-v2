2.4. Encapsulation & Immutability

Encapsulation (Đóng gói) không chỉ là "giấu dữ liệu". Nó là việc nhóm dữ liệu và các hành động trên dữ liệu đó vào một đơn vị, đồng thời kiểm soát chặt chẽ cách thức thế giới bên ngoài tương tác với dữ liệu đó để tránh trạng thái không hợp lệ (Inconsistent State).

A Access Modifiers: Ma trận quyền truy cập

Modifier Class Package Subclass World
private ✅❌❌❌
default (no modifier) ✅✅❌❌
protected ✅✅✅❌
public ✅✅✅✅
Lời khuyên của Thầy:

Luôn bắt đầu với private. Chỉ nới lỏng khi thực sự cần thiết. Quyền truy cập càng hẹp, hệ thống càng ít lỗi "side-effect".

Cảnh báo Protected:

Protected cho phép các class cùng package truy cập, dù không kế thừa. Đây là một "lỗ hổng" thiết kế cần cẩn trọng.

F Từ khóa FINAL: Chìa khóa của sự bất biến

Final Variable

Chỉ được gán giá trị 1 lần duy nhất. Giúp đảm bảo Thread-Safety trong lập trình đa luồng.

Final Method

Không thể bị Override ở lớp con. Ngăn chặn việc làm thay đổi logic cốt lõi của class cha.

Final Class

Không thể bị kế thừa. Ví dụ tiêu biểu là String. Giúp đảm bảo an ninh và tối ưu hóa bộ nhớ.

// Một Immutable Class mẫu mực

public final class UserAccount {
    private final String username; // Immutable
    private final List<String> roles; // Phải cẩn thận với Mutable objects

    public UserAccount(String username, List<String> roles) {
        this.username = username;
        // Defensive Copy: Senior tip để bảo mật dữ liệu
        this.roles = List.copyOf(roles); 
    }

    public String getUsername() { return username; }
    
    // Không bao giờ trả về trực tiếp reference của mutable object
    public List<String> getRoles() { return roles; } 
}

Tại sao nên dùng Final cho hiệu năng?

JIT Inlining

Khi một method là final, JVM biết chắc chắn nó sẽ không bị ghi đè. JIT Compiler có thể thực hiện Method Inlining: thay thế lời gọi hàm bằng mã nguồn thực tế của hàm đó. Việc này loại bỏ chi phí đẩy tham số vào Stack và nhảy địa chỉ lệnh (Jump).

Constant Folding

Với static final, compiler có thể tính toán trước giá trị ngay từ khi biên dịch. Giúp giảm thiểu các phép tính toán không cần thiết lúc Runtime.

Senior Best Practices

01
Thiết kế cho sự bất biến (Immutability)

"Classes should be immutable unless there's a very good reason to make them mutable." (Joshua Bloch - Effective Java)

02
Defensive Copying

Khi nhận vào hoặc trả về một Mutable Object (như Date, List), hãy tạo một bản sao mới thay vì dùng chung reference để tránh bị sửa lén dữ liệu từ bên ngoài.

03
Tell, Don't Ask

Đừng lấy dữ liệu ra để xử lý (Asking), hãy ra lệnh cho đối tượng thực hiện hành động trên chính dữ liệu của nó (Telling).

Góc Senior Interview

Q1: Từ khóa "final" có ý nghĩa gì đối với một Reference Variable trỏ tới một List? ▼
Nó có nghĩa là biến tham chiếu đó không thể trỏ tới một List khác. Tuy nhiên, nội dung bên trong List đó vẫn có thể bị thay đổi (add, remove). Để làm cho List thực sự bất biến, ta phải dùng Collections.unmodifiableList() hoặc List.copyOf().
Q2: Tại sao các hằng số thường được khai báo là "public static final"? ▼
- public: Để mọi nơi có thể truy cập.
- static: Để biến đó thuộc về class, không tốn bộ nhớ cho từng instance.
- final: Để giá trị không bao giờ bị thay đổi.
