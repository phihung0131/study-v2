2.2. Interface Evolution

Khám phá hành trình lột xác của Interface từ Java 7 trở về trước cho đến những cải tiến mạnh mẽ trong Java 8 và 9+.

🕒 Lịch sử tiến hóa qua các phiên bản

Java 7 & Cũ hơn: "Bản hợp đồng thuần túy"

Chỉ chứa phương thức trừu tượng (abstract) và hằng số. Mọi lớp implement đều phải viết code cho tất cả các phương thức.

Nỗi đau: Nếu thêm 1 method vào interface, hàng ngàn lớp triển khai sẽ bị lỗi biên dịch!

Java 8: "Bước ngoặt Default & Static"

Giới thiệu default methods để hỗ trợ Backward Compatibility (ví dụ thêm stream() vào List).

Java 9: "Private Methods - Đóng gói nội bộ"

Cho phép định nghĩa private methods bên trong interface để tái sử dụng code giữa các default methods mà không lộ ra ngoài.

Default Methods: "Phao cứu sinh" cho API

Cú pháp default cho phép chúng ta triển khai logic ngay trong interface.

public interface SmartDevice {
    void turnOn(); // Abstract

    // Default method: Cung cấp logic mặc định
    default void diagnostic() {
        System.out.println("Running basic diagnostic...");
        checkPower();
    }

    // Java 9 Private method: Trình helper nội bộ
    private void checkPower() {
        System.out.println("Power is stable.");
    }
}
Tại sao cần Default Methods?

Hãy tưởng tượng Oracle muốn thêm phương thức forEach vào interface Iterable. Nếu không có default, hàng triệu ứng dụng Java trên thế giới sẽ không thể chạy được sau khi nâng cấp vì lỗi "Class không implement method mới".

Xử lý xung đột: Bài toán "Kim cương"

Khi một lớp implement 2 interface có cùng một phương thức default, Java sẽ xử lý như thế nào?

Interface A
default void hello()
Interface B
default void hello()
⚠️ LỖI BIÊN DỊCH: Duplicate default methods!

Cách giải quyết (Rule of Thumb):

  • Class luôn thắng: Nếu class cha đã implement method đó, interface bị bỏ qua.
  • Ghi đè thủ công: Subclass phải ghi đè (override) và chỉ định rõ muốn dùng của ai.
public class MyClass implements A, B {
    @Override
    public void hello() {
        // Tự quyết định: gọi của A
        A.super.hello(); 
    }
}

JVM Internal: invokeinterface vs invokevirtual

// Khi gọi một default method:

1. JVM kiểm tra xem Class thực tế có override không.

2. Nếu không, nó tìm trong cây phân cấp Interface.

3. Nó sử dụng một bảng gọi là ITable để tra cứu.

Deep Note: Mặc dù invokeinterface từng chậm hơn invokevirtual, nhưng nhờ Monomorphic Inlining của JIT Compiler, hiệu năng hiện nay gần như tương đương.

Mục lục Senior Interview

Tại sao Interface không có biến instance?

Vì Interface định nghĩa hành vi, không định nghĩa trạng thái. Nếu có biến instance, nó sẽ trở thành "Đa kế thừa trạng thái" - thứ mà Java kịch liệt tránh để ngăn chặn sự phức tạp của bộ nhớ.

Sự khác biệt giữa Static method trong Interface và Class?

Static method trong Interface không được kế thừa bởi class implement. Em phải gọi trực tiếp qua tên Interface (InterfaceName.methodName()).
