2.3. Polymorphism

Polymorphism (Đa hình) là khả năng một đối tượng có thể thực hiện một hành động theo nhiều cách khác nhau. Trong Java, điều này cho phép một biến tham chiếu kiểu lớp cha trỏ đến đối tượng của lớp con.

Compile-time Polymorphism

Thực hiện qua Method Overloading. Quyết định gọi hàm nào dựa trên tham số truyền vào lúc biên dịch.

Runtime Polymorphism

Thực hiện qua Method Overriding. Quyết định gọi hàm nào dựa trên kiểu đối tượng thực tế tại Runtime.

2 Static vs Dynamic Binding

// Phân tích Binding qua Code

public class BindingDemo {
    public static void main(String[] args) {
        Animal myDog = new Dog();

        // 1. Static Binding (Compile-time)
        // JVM nhìn vào kiểu khai báo (Animal)
        System.out.println(myDog.species); 

        // 2. Dynamic Binding (Runtime)
        // JVM nhìn vào đối tượng thực tế (Dog)
        myDog.makeSound(); 
    }
}
Static Binding (Early Binding)
  • ⚡ Xảy ra khi biên dịch (Compile-time).
  • 🛠️ Áp dụng cho: static, private, final methods và toàn bộ variables.
  • 🚀 Hiệu năng cực cao vì địa chỉ hàm đã xác định sẵn.
Dynamic Binding (Late Binding)
  • 🔄 Xảy ra khi thực thi (Runtime).
  • 🛠️ Áp dụng cho các virtual methods (Overriding).
  • 🔎 Cần cơ chế tra cứu (Lookup) địa chỉ hàm.

Cơ chế V-Table (Virtual Method Table)

Làm sao JVM biết myDog.makeSound() phải chạy code của lớp Dog chứ không phải Animal? Câu trả lời là V-Table.

Mô phỏng V-Table trong Bộ nhớ

Class Animal
[0] eat() @Animal_Eat_Addr
[1] makeSound() @Animal_Sound_Addr
⬇️ Inheritance ⬇️
Class Dog extends Animal
[0] eat() @Animal_Eat_Addr
[1] makeSound() @Dog_Sound_Addr (Overridden)
Quy trình thực thi:
  1. 1 JVM tải class Dog, tạo một V-Table chứa danh sách địa chỉ các method.
  2. 2 Nếu Dog override makeSound(), địa chỉ trong slot [1] sẽ được cập nhật trỏ tới code của Dog.
  3. 3 Khi gọi myDog.makeSound(), JVM tìm đến V-Table của đối tượng thực tế (Dog) và lấy địa chỉ ở slot tương ứng.
"V-Table giúp việc tra cứu method đạt độ phức tạp O(1), cực kỳ nhanh!"

Method Hiding & Variable Shadowing

Cảnh báo lỗi phổ biến:

Đa hình chỉ áp dụng cho Instance Methods. Nó không áp dụng cho Variables và Static Methods.

class Parent {
    String name = "Parent";
    static void display() { System.out.println("Parent Static"); }
}

class Child extends Parent {
    String name = "Child"; // Variable Shadowing
    static void display() { System.out.println("Child Static"); } // Method Hiding
}

// Test
Parent p = new Child();
System.out.println(p.name); // Output: Parent
p.display();               // Output: Parent Static
Tại sao kết quả lại là "Parent"?

Vì Variables và Static Methods sử dụng Static Binding. Compiler quyết định dựa trên kiểu của biến tham chiếu (Parent p), không quan tâm đối tượng thực sự là gì.

Đây là lý do tại sao ta không bao giờ nên định nghĩa lại biến hoặc static method trùng tên trong lớp con.

Góc Senior Interview

Q: Có thể gọi một method của subclass từ tham chiếu của superclass mà không cần Downcasting không?

A: Không. Compiler kiểm tra tính hợp lệ dựa trên kiểu tham chiếu. Nếu method đó không được khai báo trong superclass, compiler sẽ báo lỗi ngay cả khi đối tượng thực tế có method đó. Ta phải dùng Downcasting (ép kiểu xuống).

Q: Covariant Return Type là gì? Nó liên quan gì đến Đa hình?

A: Cho phép lớp con ghi đè một method và trả về một kiểu dữ liệu là subclass của kiểu dữ liệu ở lớp cha. Điều này giúp code linh hoạt hơn mà vẫn đảm bảo tính đa hình.
