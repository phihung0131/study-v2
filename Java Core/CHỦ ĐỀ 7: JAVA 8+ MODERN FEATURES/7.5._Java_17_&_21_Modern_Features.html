1. Records

Java 16/17+

Tạm biệt boilerplate code (Getter, Equals, HashCode, ToString). Records là các class Immutability mặc định dùng để mang dữ liệu.

// Chỉ 1 dòng duy nhất!
public record User(Long id, String name, String email) {}

// Cách dùng
User user = new User(1L, "Thầy Java", "mentor@java.com");
System.out.println(user.name()); // Không dùng getName()
Under the hood & Rules:
  • • Mặc định là final, không thể kế thừa.
  • • Các field là private final.
  • • Có thể thêm Compact Constructor để validate dữ liệu.
  • • Tuyệt vời cho DTOs, API Responses, Keys trong Maps.

2. Sealed Classes

Java 17+

Cho phép em kiểm soát chính xác ai được phép kế thừa class của mình. Đây là nền tảng cho Algebraic Data Types trong Java.

public sealed interface Shape 
    permits Circle, Rectangle, Square {}

public final class Circle implements Shape { ... }
public final class Square implements Shape { ... }
// Bất kỳ class nào khác cố tình implements Shape sẽ bị lỗi compile!
Senior Tip: Khi kết hợp với Switch (Pattern Matching), trình biên dịch có thể kiểm tra xem em đã xử lý hết các case chưa (Exhaustiveness check), giúp loại bỏ lỗi logic.

3. Pattern Matching & Switch

Java 21+

Switch bây giờ không chỉ kiểm tra giá trị, mà còn kiểm tra được cả Kiểu dữ liệu (Type) và Cấu trúc (Record Patterns).

static String formatter(Object obj) {
    return switch (obj) {
        case Integer i -> String.format("Số nguyên: %d", i);
        case String s  -> String.format("Chuỗi: %s", s);
        case User(var id, var name, var email) -> "Chào " + name; // Record Pattern
        case null      -> "Rỗng rồi em ơi!";
        default        -> obj.toString();
    };
}

4. Virtual Threads (Project Loom)

Java 21+ HOT

Đây là tính năng quan trọng nhất trong vòng 10 năm qua. Virtual Threads là các luồng siêu nhẹ, em có thể tạo hàng triệu luồng mà không lo tràn RAM (Stack size của Virtual Thread cực nhỏ và linh hoạt).

// Tạo Executor chạy Virtual Threads
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i -> {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        });
    });
} // Tự động đóng và chờ hoàn thành
Bản chất: Virtual Threads được JVM quản lý, không phải OS. Khi một Virtual Thread gặp I/O chặn (Blocking), JVM "unmount" nó khỏi Platform Thread để nhường chỗ cho Virtual Thread khác chạy.

5. Sequenced Collections

Java 21+

Sửa lỗi thiết kế "nghìn năm" của Java Collections: Thiếu một interface chung để xử lý thứ tự từ đầu đến cuối.

List<String> list = List.of("A", "B", "C");
list.getFirst(); // "A"
list.getLast();  // "C"
list.reversed(); // Trả về view đảo ngược cực nhanh

🎯 Senior Interview Mastery

Q1: Tại sao Virtual Threads lại làm cho Web Servers (như Tomcat/Jetty) mạnh hơn?

Trước đây, mỗi Request tốn 1 Platform Thread (thường tốn 1MB RAM). Với Virtual Threads, chúng ta có thể phục vụ hàng triệu request đồng thời trên cùng một lượng phần cứng cũ, vì CPU không còn phải chờ đợi I/O một cách lãng phí (Context switch ở tầng OS cực kỳ đắt đỏ, còn ở tầng JVM thì rất rẻ).

Q2: Khi nào KHÔNG nên dùng Record?

Đừng dùng Record cho Hibernate Entities nếu em cần tính năng Lazy Loading (Hibernate yêu cầu class không được final và có constructor rỗng). Record cũng không phù hợp nếu em cần thay đổi trạng thái (Mutable) của đối tượng sau khi khởi tạo.

Q3: Sealed Classes có giúp ích gì cho Performance không?

Có. Vì JVM biết chắc chắn danh sách các subclass tại thời điểm biên dịch, JIT compiler có thể thực hiện tối ưu hóa tốt hơn như **Inlining** hoặc loại bỏ các kiểm tra kiểu không cần thiết.
