Vấn nạn "Exception Masking"

Hãy tưởng tượng: Luồng xử lý chính của em bị lỗi (ví dụ: DB sập). Trong khối finally, em cố gắng đóng kết nối nhưng việc đóng kết nối cũng bị lỗi. Trong Java < 7, lỗi đóng kết nối sẽ "nuốt chửng" lỗi DB ban đầu.

Primary Error (DB)
Secondary Error (Close)

Kết quả:

  • ➜ Masking: Em chỉ thấy "Secondary Error". Manh mối về vụ DB sập biến mất hoàn toàn.
  • ➜ Suppression: Em thấy "Primary Error", và bên trong nó có chứa thông tin về "Secondary Error".

2. Suppressed Exception là gì?

Suppressed Exception (Ngoại lệ bị nén) là một cơ chế cho phép một Exception "mang theo" các Exception khác xảy ra sau nó nhưng cùng trong một ngữ cảnh xử lý.

Quy tắc của Try-with-resources:

  1. Nếu khối try ném lỗi, lỗi này là Primary.
  2. Khi tự động gọi close(), nếu có lỗi phát sinh, nó sẽ được đánh dấu là Suppressed.
  3. Nếu khối try KHÔNG lỗi, nhưng close() lỗi, thì lỗi của close() sẽ trở thành lỗi chính ném ra ngoài.

3. The Throwable API

Từ Java 7, lớp java.lang.Throwable được bổ sung 2 phương thức cực kỳ quan trọng:

void addSuppressed(Throwable exception)

Thêm một ngoại lệ vào danh sách "bị nén". Thường do trình biên dịch gọi tự động trong Try-with-resources.

Throwable[] getSuppressed()

Trả về một mảng chứa tất cả các ngoại lệ đã bị nén. Nếu không có, trả về mảng rỗng.

4. Manual Suppression (Làm thủ công)

Dù hiếm, nhưng đôi khi em muốn tự mình triển khai cơ chế này (ví dụ: khi không dùng TWR hoặc viết framework).

public void manualExample() throws Exception {
    Exception primary = null;
    try {
        throw new Exception("Lỗi chính: DB Connection Fail");
    } catch (Exception e) {
        primary = e;
        throw e;
    } finally {
        Exception secondary = new Exception("Lỗi phụ: Close Resource Fail");
        if (primary != null) {
            // "Gắn" lỗi phụ vào lỗi chính
            primary.addSuppressed(secondary);
        } else {
            throw secondary;
        }
    }
}

5. Deep Dive: Cơ chế lưu trữ & Performance

Dữ liệu được lưu ở đâu?

Bên trong Throwable, có một trường private:
private List<Throwable> suppressedExceptions;

Sự thật thú vị: Danh sách này được khởi tạo theo kiểu Lazy (Lười biếng). Nó chỉ chiếm bộ nhớ khi có lỗi bị nén đầu tiên được thêm vào qua addSuppressed().

Performance:

Việc nén lỗi không tốn thêm chi phí tạo StackTrace cho lỗi phụ (vì lỗi phụ đã được tạo sẵn). Tuy nhiên, việc in StackTrace ra log sẽ lâu hơn một chút vì JVM phải duyệt qua cây các ngoại lệ bị nén.

6. Câu hỏi "Hại não" cho Senior

Q1: Một Exception có thể tự nén chính nó không? (self-suppression)

Ví dụ: e.addSuppressed(e);.
Đáp án: Không. Java sẽ ném ra IllegalArgumentException để tránh vòng lặp vô tận khi in log.

Q2: Nếu em truyền `null` vào `addSuppressed`, chuyện gì xảy ra?

Đáp án: Nó cũng ném NullPointerException. Java rất khắt khe với tính toàn vẹn của Exception chain.

Q3: Làm sao để vô hiệu hóa hoàn toàn khả năng Suppression của một Exception?

Đáp án: Khi viết Custom Exception, em có thể gọi constructor của Throwable với tham số enableSuppression = false. Khi đó, mọi lời gọi addSuppressed sẽ bị lờ đi.
