Tại sao phải tạo Exception riêng?

Java cung cấp rất nhiều Exception, nhưng chúng mang tính kỹ thuật (Technical). Trong dự án thực tế, chúng ta cần những Exception mang tính nghiệp vụ (Business).

Cách làm nghiệp dư:

Dùng RuntimeException cho mọi thứ.

throw new RuntimeException("Số dư không đủ");

Người gọi (Caller) không biết lỗi gì để xử lý riêng biệt (catch block chung chung).

Cách làm Senior:

Tạo domain-specific exceptions.

throw new InsufficientFundsException(amount, balance);

Caller có thể catch riêng lỗi này để gợi ý khách hàng nạp thêm tiền.

2. Cách tạo chuẩn Senior

Một Custom Exception tốt nên hỗ trợ đầy đủ các constructor của lớp cha để giữ nguyên tính kế thừa (StackTrace, Cause).

/**
 * Base Exception cho toàn bộ ứng dụng (Unchecked)
 */
public class BaseBusinessException extends RuntimeException {
    private final String errorCode;

    public BaseBusinessException(String message, String errorCode) {
        super(message);
        this.errorCode = errorCode;
    }

    public BaseBusinessException(String message, String errorCode, Throwable cause) {
        super(message, cause); // Quan trọng: Giữ lại "nguyên nhân gốc"
        this.errorCode = errorCode;
    }

    public String getErrorCode() { return errorCode; }
}

Lời khuyên: Hầu hết các dự án hiện đại (Spring Boot, Microservices) ưu tiên kế thừa từ RuntimeException để tránh việc "ô nhiễm" phương thức bởi từ khóa throws.

3. Exception Payload (Metadata)

Đừng chỉ gửi Message string. Hãy gửi dữ liệu để UI hoặc System log có thể xử lý thông minh hơn.

Ví dụ: OTP Invalid Exception

public class InvalidOtpException extends BaseBusinessException {
    private final int attemptsLeft;
    private final LocalDateTime expiryTime;

    public InvalidOtpException(int left, LocalDateTime expiry) {
        super("Mã OTP không đúng", "ERR_OTP_001");
        this.attemptsLeft = left;
        this.expiryTime = expiry;
    }
    // Getters...
}

Lợi ích:

  • ✅ Tự động hóa việc hiển thị UI (Số lần thử còn lại).
  • ✅ Phân loại lỗi theo Code để quốc tế hóa (i18n).
  • ✅ Giúp log trace-id dễ dàng hơn.

4. Optimization: "Fast" Exceptions

Như bài trước thầy đã dạy, fillInStackTrace() là phần nặng nhất. Nếu Exception chỉ mang tính chất báo hiệu (Signaling) và em không cần log stack trace, hãy tắt nó đi.

public class FastBusinessException extends RuntimeException {
    public FastBusinessException(String message) {
        // Tham số cuối: writableStackTrace = false
        super(message, null, true, false);
    }
}
Exception này nhanh gần bằng việc trả về một giá trị thông thường (return).

5. Danh sách Best Practices

✔

Hậu tố "Exception"

Luôn kết thúc tên class bằng chữ "Exception".

✖

Tránh "Exception Pollution"

Đừng tạo quá nhiều Exception nhỏ lẻ. Hãy gom nhóm theo module.

✔

Cung cấp Cause

Luôn truyền lỗi gốc (cause) vào constructor nếu đang wrap lỗi.

✔

Serializable

Exceptions là Serializable. Hãy thêm serialVersionUID.

6. Câu hỏi phỏng vấn "Hóc búa"

Q1: Tại sao Joshua Bloch (tác giả Effective Java) khuyên nên cung cấp các phương thức truy cập (Accessors) cho dữ liệu trong Custom Exception?

Đáp án: Để cho phép chương trình phục hồi (Programmatic recovery). Nếu Exception chỉ có string message, code xử lý lỗi sẽ phải parse string (rất tệ) để lấy dữ liệu.

Q2: Khi nào thì KHÔNG nên tạo Custom Exception?

Đáp án: Khi các exception tiêu chuẩn của Java đã mô tả hoàn hảo lỗi đó (vd: `NullPointerException` hay `IllegalArgumentException`). Đừng cố "re-invent the wheel" nếu không có thêm thông tin nghiệp vụ đặc thù.
