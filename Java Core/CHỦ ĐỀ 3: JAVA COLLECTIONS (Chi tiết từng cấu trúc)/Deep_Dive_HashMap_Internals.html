HashMap Internals

Học cách Java quản lý hàng triệu cặp Key-Value với độ phức tạp O(1) thông qua Hashing, Bitwise và Red-Black Trees.

1 Kiến trúc Node & Table

HashMap thực chất là một Mảng của các Node. Mỗi Node chứa 4 thông tin cốt lõi:

  • hash: Kết quả của hàm hash (int 32-bit).
  • key: Khóa (cho phép null).
  • value: Giá trị tương ứng.
  • next: Con trỏ tới Node tiếp theo (LinkedList) hoặc liên kết trong Tree. 
// Cấu trúc cơ bản trong Java 17
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next; // Chỉ dùng khi có Collision
}

// Bảng chứa chính
transient Node<K,V>[] table;

2 Hashing Logic: XOR & Shift

Để giảm thiểu Collision, Java thực hiện một bước "xáo trộn" các bit của `hashCode()` gọi là Perturbation Function.

Hashing
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

Tại sao lại dùng `h ^ (h >>> 16)`?

Các bảng băm thường có kích thước nhỏ. Nếu chỉ dùng các bit thấp của hashCode, ta sẽ mất thông tin từ các bit cao. Phép XOR này giúp "kéo" sức ảnh hưởng của các bit cao xuống các bit thấp, giúp phân bổ các phần tử vào bucket đều hơn, giảm Collision đáng kể.

3 Cách tính Index (Vũ điệu bitwise)

Thông thường để tìm index trong mảng kích thước `n`, ta dùng phép chia lấy dư: `hash % n`.

Nhưng trong HashMap, vì n luôn là lũy thừa của 2, Java dùng phép toán bitwise cực nhanh:

index = (n - 1) & hash

* Phép & nhanh hơn rất nhiều so với phép chia % ở mức độ CPU.

Ví dụ với n = 16:

(n-1) = 15: 0000 0000 0000 1111
Hash   : 1011 0101 1101 1010
Result : 0000 0000 0000 1010 (Index 10)

4 Collision: LinkedList vs Red-Black Tree

Khi hai key khác nhau có cùng index, chúng sẽ "đánh nhau" (Collision). Java 8+ xử lý việc này cực kỳ tinh tế:

Giai đoạn 1: LinkedList

Mặc định, các Node trùng index sẽ nối đuôi nhau thành một danh sách liên kết. Tìm kiếm tốn O(n).

Giai đoạn 2: Tree-fication

Khi số Node trong 1 bucket đạt ngưỡng 8 và tổng size table >= 64, LinkedList sẽ biến thành Red-Black Tree. Tìm kiếm tốn O(log n).

Tại sao lại là con số 8?

Dựa trên phân phối Poisson, xác suất để 8 phần tử rơi vào cùng một bucket là cực kỳ thấp (khoảng 0.00000006). Tree-fication giống như một "phao cứu sinh" để chống lại các cuộc tấn công Hash DoS.

6 Resize & Transfer

Khi số lượng phần tử vượt quá threshold (capacity * loadFactor), HashMap sẽ tự nhân đôi kích thước.

  • Capacity mới = 2 * Capacity cũ.
  • Toàn bộ phần tử phải được Re-hashed (tính lại index).

Điểm hay của lũy thừa 2:

Khi resize, index của một phần tử chỉ có 2 khả năng: giữ nguyên vị trí cũ hoặc chuyển sang vị trí cũ + oldCapacity. Java sử dụng bitwise để kiểm tra điều này cực nhanh mà không cần tính lại hash thực sự.

Senior Best Practices

Sử dụng Immutable Key

Tốt nhất là `String` hoặc `Integer`. Nếu Key thay đổi trạng thái sau khi put, `hashCode` thay đổi -> Em sẽ không bao giờ lấy lại được Value đó nữa (Memory Leak!).

Initial Capacity

Nếu biết trước Map sẽ chứa 1000 phần tử, hãy khởi tạo: new HashMap<>(1000 / 0.75 + 1). Điều này ngăn chặn việc resize liên tục tốn tài nguyên.

Thử thách Mentor (Interview)

Tại sao HashMap lại không an toàn trong môi trường đa luồng? ▼

1. Race condition: Hai luồng cùng put vào một bucket có thể làm mất dữ liệu của nhau.

2. Infinite Loop (Java 7): Trong quá trình resize, cơ chế đảo ngược LinkedList có thể gây ra vòng lặp vô tận (đã được fix trong Java 8 nhưng vẫn không an toàn).

3. Fail-fast: Iterator sẽ ném ConcurrentModificationException nếu Map bị sửa đổi cấu trúc trong lúc duyệt.

Nếu tất cả Key đều có cùng hashCode() thì sao? ▼

Mọi phần tử sẽ rơi vào duy nhất một bucket. HashMap sẽ bị thoái hóa thành một LinkedList (Java 7) với hiệu năng O(n), hoặc một Red-Black Tree (Java 8+) với hiệu năng O(log n). Tốc độ của Map sẽ chậm đi đáng kể.
