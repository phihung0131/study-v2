Fail-fast vs. Fail-safe

Hiểu sâu cơ chế bảo vệ dữ liệu của Java khi duyệt qua các tập hợp trong môi trường đơn luồng và đa luồng.

1 Triết lý Thiết kế

Fail-fast: "Thà hy sinh còn hơn sai sót"

Khi một Iterator nhận thấy cấu trúc của tập hợp bị thay đổi (thêm/xóa), nó sẽ ném lỗi ngay lập tức. Mục tiêu là ngăn chặn việc duyệt tiếp trên một trạng thái không ổn định, tránh dẫn đến các lỗi khó tìm (non-deterministic) về sau.

Fail-safe: "Tiếp tục công việc bất chấp"

Còn được gọi là Weakly Consistent. Iterator làm việc trên một bản sao hoặc snapshot của dữ liệu. Nó không bao giờ ném lỗi khi cấu trúc gốc bị thay đổi, nhưng đổi lại, nó có thể không thấy được các cập nhật mới nhất.

2 Fail-fast & Vũ điệu modCount

// Bên trong ArrayList.java (Mô phỏng)

public class ArrayList<E> {
    protected transient int modCount = 0; // Đếm số lần thay đổi cấu trúc

    public void add(E e) {
        modCount++; // Tăng mỗi khi add
        // ... logic add ...
    }

    private class Itr implements Iterator<E> {
        int expectedModCount = modCount; // Chốt số tại thời điểm tạo Iterator

        public E next() {
            checkForComodification();
            // ... logic next ...
        }

        final void checkForComodification() {
            if (modCount != expectedModCount)
                throw new ConcurrentModificationException();
        }
    }
}

Lưu ý: Fail-fast không chỉ xảy ra trong đa luồng. Ngay cả trong đơn luồng, nếu em vừa duyệt vừa gọi list.remove() thay vì iterator.remove(), lỗi này vẫn sẽ xảy ra.

3 Fail-safe (Weakly Consistent)

Các Collection trong package java.util.concurrent (như CopyOnWriteArrayList, ConcurrentHashMap) sử dụng chiến thuật này.

Cơ chế Copy-On-Write:

Khi Iterator được tạo, nó giữ một tham chiếu đến mảng hiện tại (Snapshot). Nếu có luồng nào thêm phần tử, một mảng MỚI hoàn toàn sẽ được tạo ra, còn Iterator cũ vẫn tiếp tục duyệt trên mảng CŨ.

// CopyOnWriteArrayList.java
final Object[] elements = getArray();
return new COWIterator<E>(elements, 0);

// Bất kể dữ liệu gốc thay đổi thế nào
// Iterator này vẫn trung thành với 'elements' cũ

5 Bảng so sánh Chi tiết

Tiêu chí Fail-fast Fail-safe (Weakly Consistent)
Exception Ném ConcurrentModificationException Không bao giờ ném Exception
Cơ chế duyệt Duyệt trực tiếp trên dữ liệu gốc Duyệt trên bản sao/snapshot hoặc cấu trúc concurrent
Hiệu năng bộ nhớ Tốt (không tốn thêm bộ nhớ) Tốn thêm (do tạo bản sao - CopyOnWrite)
Ví dụ tiêu biểu ArrayList, HashMap, HashSet, Vector CopyOnWriteArrayList, ConcurrentHashMap

6 Senior Best Practices

Dùng Iterator.remove()

Nếu cần xóa phần tử khi đang duyệt trong đơn luồng, hãy dùng iterator.remove(). Nó sẽ cập nhật expectedModCount cho khớp với modCount.

Đừng tin tuyệt đối Fail-safe

Nhớ rằng Fail-safe có tính "Weakly Consistent". Nếu một luồng khác vừa add phần tử vào, Iterator của em có thể sẽ KHÔNG thấy nó. Đừng dùng nó cho các logic cần tính realtime tuyệt đối.

Java 8 RemoveIf

Thay vì viết loop rườm rà, hãy dùng list.removeIf(e -> condition). Nó an toàn, ngắn gọn và đã được tối ưu bên trong.

Câu hỏi Phỏng vấn Hóc búa

Q1: Tại sao Fail-fast lại được thiết kế? Nó có bảo vệ được Thread-safety không?

Mentor Answer: KHÔNG. Fail-fast không đảm bảo thread-safety. Nó chỉ là một cơ chế phát hiện lỗi (detection) chứ không phải cơ chế phòng ngừa (prevention). Nó được thiết kế để "Fail early, Fail loudly" giúp Developer phát hiện các bug về concurrency sớm hơn.

Q2: Enumeration của Vector là Fail-fast hay Fail-safe?

Mentor Answer: Đây là câu hỏi bẫy! Enumeration là di sản từ Java 1.0, nó KHÔNG phải Fail-fast. Nếu em dùng Enumeration để duyệt Vector và sửa Vector, nó sẽ không ném Exception. Chỉ có Iterator của Vector mới là Fail-fast.
