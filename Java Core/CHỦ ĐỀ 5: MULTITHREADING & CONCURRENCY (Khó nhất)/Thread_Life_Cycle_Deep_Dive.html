Phần 5: Concurrency

Mô hình Vòng đời của Thread (Thread Life Cycle)

Trong Java, một Thread không chỉ đơn thuần là "chạy" hay "không chạy". Nó trải qua một chu kỳ sống phức tạp được quản lý bởi JVM và OS Scheduler. Hiểu rõ các trạng thái này là chìa khóa để debug các vấn đề về hiệu năng và tắc nghẽn hệ thống.

# Mô phỏng chuyển đổi trạng thái

NEW
RUNNABLE
BLOCKED
WAITING
TERMINATED
T

6 Trạng thái trong Enum Thread.State

1. NEW (Mới tạo)

Thread đã được khởi tạo bằng toán tử new nhưng phương thức start() chưa được gọi.

Thread t = new Thread(() -> {});
System.out.println(t.getState()); // NEW

2. RUNNABLE (Sẵn sàng/Đang chạy)

Sau start(). Java không phân biệt giữa "đang chạy" (Running) và "đợi CPU" (Ready).

Lưu ý Mentor: Trạng thái này có thể bị ngắt quãng bởi OS Scheduler (Time-slicing).

3. BLOCKED (Bị chặn)

Đang đợi để lấy một Monitor Lock (synchronized) để vào một block hoặc phương thức.

4. WAITING (Chờ vô thời hạn)

Thread đợi tín hiệu từ Thread khác qua wait() hoặc join().

5. TIMED_WAITING (Chờ có thời hạn)

Giống WAITING nhưng tự động thức dậy sau một khoảng thời gian: sleep(ms), wait(ms).

6. TERMINATED (Kết thúc)

Thread đã hoàn thành công việc hoặc bị kết thúc do Exception không được xử lý.

Deep Dive: JVM Internals & Memory

  • 1
    Stack Memory: Khi một Thread ở bất kỳ trạng thái nào trừ TERMINATED, nó luôn chiếm một lượng bộ nhớ Stack (mặc định ~1MB). 1000 threads = 1GB RAM chỉ để giữ Stack!
  • 2
    Context Switching: Khi Thread chuyển từ RUNNABLE sang BLOCKED/WAITING, CPU phải thực hiện "Context Switch" - lưu trạng thái thanh ghi và nạp Thread khác. Đây là thao tác đắt đỏ (Costly).
  • 3
    OS Thread Mapping: Trong Java hiện đại, 1 Java Thread thường tương ứng trực tiếp với 1 Native OS Thread (1:1 model).

Lỗi "Chết Người" (Anti-patterns)

❌ Gọi run() thay vì start()

Gọi t.run() sẽ chạy logic trên Thread hiện tại (Main), không tạo Thread mới. State vẫn là NEW!

❌ Start lại Thread đã chết

Một Thread khi đã TERMINATED không bao giờ có thể quay lại. Gọi start() lần nữa sẽ ném IllegalThreadStateException.

Câu hỏi Mentor (Interview level)

Q1: Sự khác biệt bản chất giữa BLOCKED và WAITING?

Mentor's Answer: BLOCKED là sự bị động (đợi Lock từ người khác), còn WAITING là chủ động (nhường CPU và đợi tín hiệu thông qua notify/join).

Q2: Tại sao stop() bị deprecated và không nên dùng?

Mentor's Answer: stop() giết thread ngay lập tức, khiến các Monitor Lock không được giải phóng đúng cách hoặc tài nguyên bị bỏ dở giữa chừng gây corrupt dữ liệu. Hãy dùng Flag hoặc interrupt().
