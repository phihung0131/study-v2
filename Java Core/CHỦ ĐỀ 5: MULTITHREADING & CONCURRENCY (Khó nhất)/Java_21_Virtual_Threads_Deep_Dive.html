Virtual Threads:
Cuộc Cách Mạng Về Quy Mô

Trong Java truyền thống, 1 Java Thread = 1 OS Thread. OS Thread rất đắt (1MB stack). Với Virtual Threads, bạn có thể tạo 1.000.000 luồng trên một chiếc Laptop bình thường.

01. Bức tường "Thread-per-request"

Hạn chế của Platform Threads

  • Memory: Mỗi thread tốn ~1MB RAM. 1000 threads = 1GB RAM chỉ để giữ luồng.
  • Context Switch: Việc chuyển đổi giữa các OS Thread tốn hàng ngàn chu kỳ CPU.
  • Scalability: Server thường sập khi vượt quá vài ngàn luồng.

Mô hình cũ: Chặn (Blocking) là kẻ thù của hiệu năng.

02. Lập trình với Virtual Threads

Java 21 giữ nguyên sự đơn giản của mã nguồn đồng bộ (Synchronous) nhưng mang lại sức mạnh bất đồng bộ (Asynchronous).

// Cách 1: Tạo trực tiếp
Thread vt = Thread.ofVirtual().name("my-vt").start(() -> {
    System.out.println("Running on: " + Thread.currentThread());
});

// Cách 2: Sử dụng ExecutorService (Khuyên dùng)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        // Task xử lý I/O nặng
        fetchDataFromDB();
    });
} // Tự động đóng và đợi task hoàn thành

Cảnh báo: Hiện tượng Thread Pinning

Mặc dù Virtual Threads rất mạnh, nhưng có 2 trường hợp nó sẽ "khóa chặt" (Pin) vào Carrier Thread, khiến hệ thống mất khả năng scale:

  • Pin Case 1
    Sử dụng synchronized: Nếu bạn gọi blocking I/O bên trong block synchronized, Virtual Thread không thể nhả Carrier Thread.
    -> Giải pháp: Dùng ReentrantLock thay thế.
  • Pin Case 2
    Native Methods: Khi thread đang thực thi code C/C++ (JNI).

Senior Best Practices

❌ Đừng Pool Virtual Threads

Virtual Threads cực kỳ rẻ. Việc pooling chúng chỉ làm tăng độ phức tạp và tốn memory vô ích. Hãy cứ new mỗi khi có task!

✅ Dùng cho I/O Bound

Phù hợp cho Database, API call, File system. Không dùng cho CPU-intensive (mã hóa, render video) vì Virtual Thread không giúp CPU nhanh hơn.

✅ Giới hạn tài nguyên bằng Semaphores

Vì bạn có thể tạo hàng triệu thread, hãy cẩn thận kẻo làm sập Database. Dùng Semaphore để giới hạn số thread cùng lúc truy cập DB.

✅ ThreadLocal an toàn

Virtual Threads hỗ trợ ThreadLocal, nhưng hãy cẩn thận với lượng dữ liệu lớn vì số lượng thread có thể lên tới hàng triệu.

Câu hỏi Mentor (Senior Interview)

Q: Virtual Thread khác gì với Reactive Programming (WebFlux)?

A: Cả hai đều giải quyết bài toán throughput. Tuy nhiên, Reactive bắt bạn phải thay đổi toàn bộ tư duy lập trình (Mono/Flux, callback hell). Virtual Thread cho phép bạn viết code Imperative (tuần tự) quen thuộc nhưng vẫn đạt được hiệu năng tương đương Reactive.
