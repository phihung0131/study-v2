Trapping Rain Water

LeetCode #42 | Difficulty: Hard | Array, Two Pointers, DP

🌧️ Problem Summary

Given n non-negative integers representing an elevation map where the width of each bar is 1, compute how much water it can trap after raining.

Core Intuition: The amount of water trapped above any bar at index i is determined by the minimum of the tallest bars to its left and right, minus its own height.

Water[i] = max(0, min(max_left, max_right) - height[i])

Approach 1: Brute Force (Element-wise)

Intuition: "Think Locally"

For every single bar, we look all the way to the left to find the highest wall, and all the way to the right to find the highest wall. The water level above the current bar is limited by the shorter of these two walls.

Java Implementation

public int trap(int[] height) {
    int totalWater = 0;
    int n = height.length;

    for (int currentIndex = 0; currentIndex < n; currentIndex++) {
        int maxLeft = 0;
        int maxRight = 0;

        // Search left for tallest bar
        for (int left = currentIndex; left >= 0; left--) {
            maxLeft = Math.max(maxLeft, height[left]);
        }

        // Search right for tallest bar
        for (int right = currentIndex; right < n; right++) {
            maxRight = Math.max(maxRight, height[right]);
        }

        // Water trapped is min of walls minus current height
        totalWater += Math.min(maxLeft, maxRight) - height[currentIndex];
    }
    return totalWater;
}

Complexity Analysis

Time: O(n²). For each element, we scan the whole array.

Space: O(1). No extra arrays used.

Problem

We are redundantly scanning left and right for every index. We need a way to "remember" these maximums.

Approach 2: Precomputed Max (DP)

Intuition: "Think Ahead"

Instead of scanning for every index, we can pre-calculate the max left and max right for every index in two linear passes. This "memoization" allows us to calculate the water in a final third pass in constant time per index.

Java Implementation

public int trap(int[] height) {
    if (height.length == 0) return 0;
    int n = height.length;
    int[] leftMax = new int[n];
    int[] rightMax = new int[n];

    leftMax[0] = height[0];
    for (int i = 1; i < n; i++) {
        leftMax[i] = Math.max(height[i], leftMax[i - 1]);
    }

    rightMax[n - 1] = height[n - 1];
    for (int i = n - 2; i >= 0; i--) {
        rightMax[i] = Math.max(height[i], rightMax[i + 1]);
    }

    int totalWater = 0;
    for (int i = 0; i < n; i++) {
        totalWater += Math.min(leftMax[i], rightMax[i]) - height[i];
    }
    return totalWater;
}

Complexity Analysis

Time: O(n). Three independent passes.

Space: O(n). Two extra arrays of size n.

Benefit

Significant speedup! But we can actually solve this without the extra $O(n)$ space.

Approach 3: Two Pointers (Optimal)

Intuition: "The Closing In"

We maintain two pointers (left and right) and two variables (leftMax and rightMax). The water trapped at either pointer depends on the shorter of the global maxes. By moving the pointer at the shorter wall inward, we can calculate water incrementally without extra arrays.

Click Next to observe the two pointers converging.

Java Implementation (Optimal)

public int trap(int[] height) {
    int leftPointer = 0;
    int rightPointer = height.length - 1;
    int leftMaxSoFar = 0;
    int rightMaxSoFar = 0;
    int totalWater = 0;

    while (leftPointer < rightPointer) {
        if (height[leftPointer] < height[rightPointer]) {
            // Process left side because it's the limiting factor
            if (height[leftPointer] >= leftMaxSoFar) {
                leftMaxSoFar = height[leftPointer];
            } else {
                totalWater += leftMaxSoFar - height[leftPointer];
            }
            leftPointer++;
        } else {
            // Process right side
            if (height[rightPointer] >= rightMaxSoFar) {
                rightMaxSoFar = height[rightPointer];
            } else {
                totalWater += rightMaxSoFar - height[rightPointer];
            }
            rightPointer--;
        }
    }
    return totalWater;
}

Complexity Analysis

Time: O(n). Single pass from both ends.

Space: O(1). Only 4 variables used.

Key Takeaway

"The water at index i is limited by the shorter side." By always moving the shorter side, we ensure that we only care about the max wall on THAT side.

Efficiency Comparison

Approach Time Space Efficiency
Brute Force O(n²) O(1) Poor
Dynamic Programming O(n) O(n) Good
Two Pointers O(n) O(1) Optimal

⚡ Cheat Sheet

  • • Goal: Calculate water trapped at each index.
  • • Pattern: Min(MaxLeft, MaxRight) - CurrentHeight.
  • • Trick: Always process the pointer with the smaller height/max.
  • • Recall: If `leftHeight < rightHeight`, we don't need to know `rightMax` for the left index because the right side already acts as a "strong enough" boundary.

🧩 Common Edge Cases

  • [0, 1, 0]: No water trapped (slope).
  • [4, 2, 3]: 1 unit of water trapped.
  • [1, 2, 3, 4, 5]: Ascending - No water.
  • Empty Array: Should return 0.
