Majority Element

LeetCode #169 | Difficulty: Easy | Array, Greedy, Sorting

📋 Problem Statement

Given an array nums of size n, return the majority element. The majority element is the element that appears more than ⌊n / 2⌋ times. You may assume that the majority element always exists in the array.

Note: Can you solve this in O(n) time and O(1) space?

Approach 1: Brute Force (Nested Loops)

Intuition

For every element in the array, scan the entire array to count how many times that specific element appears. If the count exceeds n / 2, we've found our answer.

Java Code

public int majorityElement(int[] nums) {
    int majorityCount = nums.length / 2;

    for (int outer = 0; outer < nums.length; outer++) {
        int count = 0;
        for (int inner = 0; inner < nums.length; inner++) {
            if (nums[inner] == nums[outer]) {
                count++;
            }
        }
        if (count > majorityCount) {
            return nums[outer];
        }
    }
    return -1;
}

Complexity

Time: O(n²). Two nested loops iterate over the array.

Space: O(1). No extra data structures used.

Inefficiency

We re-count the same values multiple times. This approach is only practical for very small arrays.

Approach 2: HashMap (Frequency Tracking)

Intuition

Avoid redundant counts by storing the frequency of each element in a Map. As we iterate, we update the frequency and immediately check if it has crossed the threshold.

public int majorityElement(int[] nums) {
    Map<Integer, Integer> counts = new HashMap<>();
    int majorityCount = nums.length / 2;

    for (int num : nums) {
        // Update frequency in the map
        counts.put(num, counts.getOrDefault(num, 0) + 1);
        
        // Check if this number is now the majority
        if (counts.get(num) > majorityCount) {
            return num;
        }
    }
    return -1;
}

Complexity

Time: O(n). Single pass through the array. Map operations are O(1) on average.

Space: O(n). In the worst case, the map stores n/2 + 1 unique elements.

Pros & Cons

Pros: Simple and fast. Cons: Uses extra linear memory which can be large.

Approach 3: Boyer-Moore Voting (Optimal)

Intuition: "The Tug of War"

Imagine a battle. A candidate takes the throne. Every time we see the same person, their count increases. Every time we see an enemy, the count decreases. If the count reaches 0, the throne is empty, and the next person we see takes it. Because the majority element appears more than half the time, it will always survive the "battles."

Java Code

public int majorityElement(int[] nums) {
    int count = 0;
    Integer candidate = null;

    for (int num : nums) {
        // If count is zero, we pick a new candidate
        if (count == 0) {
            candidate = num;
        }
        
        // Increment if current number matches candidate, else decrement
        count += (num == candidate) ? 1 : -1;
    }

    return candidate;
}

Complexity

Time: O(n). Single pass, no complex operations.

Space: O(1). Only two variables needed.

Key Insight

This works only because a majority element is guaranteed to exist. It "cancels out" all other elements and still remains.

Approach Comparison

Strategy Time Space Use Case
Brute Force O(n²) O(1) Small datasets
HashMap O(n) O(n) General purpose / multiple frequencies needed
Boyer-Moore O(n) O(1) Optimal when majority is guaranteed

💡 Cheat Sheet

  • • Majority Definition: Frequency > n / 2.
  • • Sorting Hack: If you sort the array, the majority element is always at nums[n/2].
  • • Divide & Conquer: If a majority element exists in the whole array, it must be the majority element in at least one of the halves.

🧩 Edge Cases

  • Array size is 1: The element itself is the majority.
  • Only two types of numbers: Logic handles this easily.
  • Majority element at the very end: Handled by Boyer-Moore.
