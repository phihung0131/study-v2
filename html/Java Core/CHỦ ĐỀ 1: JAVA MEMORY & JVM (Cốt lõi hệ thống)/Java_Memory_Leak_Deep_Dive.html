Bài 1.7: Memory Leak trong Java

"Khi Garbage Collector trở nên bất lực."

1. Bản chất của Memory Leak trong Java

Trong các ngôn ngữ như C, rò rỉ là do bạn quên gọi free(). Trong Java, rò rỉ xảy ra khi:

  • Đối tượng không còn cần thiết cho logic nghiệp vụ.
  • Nhưng nó vẫn được tham chiếu bởi một GC Root (biến static, thread đang chạy...).

Hệ quả:

Memory usage tăng dần -> GC chạy liên tục (High CPU) -> Stop-the-world lâu -> OutOfMemoryError.

2. "Tứ đại ác nhân" gây Memory Leak

A. Lạm dụng Static Collections

Tại sao nó nguy hiểm?

Các biến static tồn tại suốt vòng đời của ứng dụng. Nếu bạn thêm dữ liệu vào một Static Map/List mà không bao giờ xóa (remove), vùng nhớ đó sẽ bị chiếm giữ vĩnh viễn.

public class CacheManager {
    // KẺ GÂY ÁN: List này sẽ phình to mãi mãi
    private static final List<Object> CACHE = new ArrayList<>();

    public static void addToCache(Object obj) {
        CACHE.add(obj); // Quên remove hoặc không có chính sách dọn dẹp
    }
}

B. Obsolete References (Tham chiếu lỗi thời)

Đây là ví dụ kinh điển từ cuốn Effective Java. Khi ta tự quản lý bộ nhớ (ví dụ tự viết cấu trúc Stack).

public Object pop() {
    if (size == 0) throw new EmptyStackException();
    Object result = elements[--size];
    // SAI LẦM: elements[size] vẫn giữ tham chiếu tới đối tượng cũ
    return result; 
    
    // SỬA LỖI: elements[size] = null; // Loại bỏ obsolete reference
}

C. Anonymous & Non-static Inner Classes

Sự thật ngầm hiểu:

Mọi Non-static Inner Class đều giữ một tham chiếu ngầm tới Outer Class. Nếu Inner Class được truyền ra ngoài (ví dụ gán cho một thread chạy ngầm), nó sẽ giữ cho Outer Class không bao giờ bị GC.

D. ThreadLocal rò rỉ

Thường gặp trong các ứng dụng Web dùng Thread Pool (như Tomcat).

Cơ chế: Một Thread không chết sau khi xử lý Request mà quay lại Pool. Nếu bạn gán dữ liệu vào ThreadLocal mà không gọi .remove(), dữ liệu của Request cũ sẽ sống mãi trong Thread đó.

3. Làm sao để "bắt tận tay" Memory Leak?

VisualVM / JConsole

Theo dõi biểu đồ Heap Real-time. Nếu đồ thị hình "răng cưa" đi lên dần, đó là dấu hiệu leak.

Heap Dump Analysis

Chụp ảnh bộ nhớ tại một thời điểm (file .hprof) và dùng Eclipse MAT để phân tích kẻ giữ nhiều memory nhất.

JProfiler / YourKit

Công cụ trả phí mạnh mẽ, cho phép xem path từ một Object ngược về GC Root.

4. Giải pháp & Tư duy phòng ngừa

Sử dụng WeakHashMap / WeakReference

Nếu bạn cần làm Cache, hãy cân nhắc WeakHashMap. Khi key của Map không còn ai tham chiếu mạnh (Strong reference), GC sẽ tự động dọn dẹp entry đó bất kể Map có đang giữ nó hay không.

Luôn đóng resource trong block finally

Sử dụng Try-with-resources (Java 7+) để đảm bảo Connections, Streams được đóng ngay lập tức.

Unregister Listeners / Callbacks

Nếu bạn thêm một Listener vào một đối tượng sống lâu (ví dụ Singleton), hãy nhớ removeListener khi không cần thiết nữa.

Interview Mastery

Q1: Sự khác biệt giữa Memory Leak trong Java và C++ là gì?

A: Trong C++, bộ nhớ bị rò rỉ là bộ nhớ "vô chủ" (không ai quản lý). Trong Java, bộ nhớ rò rỉ là bộ nhớ "có chủ" nhưng chủ nhân không dùng tới nữa và GC thì không được phép dọn (vì vẫn còn tham chiếu).

Q2: Bạn sẽ làm gì khi nhận được thông báo OutOfMemoryError trên Production?

A: 1. Kiểm tra log xem lỗi ở vùng nào (Heap, Metaspace, Stack). 2. Nếu là Heap, kích hoạt -XX:+HeapDumpOnOutOfMemoryError. 3. Tải file dump về và dùng Eclipse MAT để tìm "Leak Suspects". 4. Kiểm tra mã nguồn xung quanh các đối tượng chiếm nhiều memory nhất.

Q3: Tại sao việc sử dụng Static Member là nguyên nhân phổ biến nhất gây leak?

A: Vì Static Members thuộc về Class, mà Class (trong hầu hết trường hợp) được nạp bởi AppClassLoader và sống vĩnh viễn. Chúng là các GC Roots, nên bất cứ thứ gì chúng tham chiếu tới đều sẽ không bao giờ được GC dọn dẹp.
