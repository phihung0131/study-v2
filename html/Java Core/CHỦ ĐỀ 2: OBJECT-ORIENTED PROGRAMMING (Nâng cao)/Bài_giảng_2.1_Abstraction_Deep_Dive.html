2.1. Abstraction: Bản chất của sự tối giản

"Abstraction là nghệ thuật bỏ qua những chi tiết không quan trọng để tập trung vào cái cốt lõi."

Trong Java, Abstraction giúp ta định nghĩa **"Cái gì" (What)** một đối tượng làm, thay vì **"Làm như thế nào" (How)**. Nó che giấu sự phức tạp của mã nguồn bên dưới một interface sạch sẽ.

Cơ chế triển khai: Abstract Class vs Interface

Đặc điểm Abstract Class Interface (Java 8+)
Mục đích Định nghĩa một thực thể chung (is-a) Định nghĩa một khả năng (can-do / contract)
State (Field) Có thể có instance variables (private, protected) Chỉ có hằng số (public static final)
Đa kế thừa Không (Chỉ kế thừa 1 class) Có (Có thể implement nhiều interface)
Constructor Có (Dùng để khởi tạo state cho subclass) Không

Mã nguồn Production-Grade

/**
 * Abstract Class: Lưu trữ trạng thái và logic chung
 */
public abstract class BaseReportGenerator {
    protected final String reportTitle; // State

    public BaseReportGenerator(String reportTitle) {
        this.reportTitle = reportTitle;
    }

    // Template Method Pattern
    public final void generate() {
        printHeader();
        fetchData();
        formatBody();
        printFooter();
    }

    protected abstract void fetchData(); // Trừu tượng hoàn toàn
    protected void printHeader() {
        System.out.println("Title: " + reportTitle);
    }
}

/**
 * Interface: Định nghĩa "Khả năng" xuất dữ liệu
 */
public interface Exportable {
    void exportToS3(String bucketName);
    
    // Default method (Java 8): Tránh làm break các client cũ
    default void logExport(String path) {
        System.out.println("Exporting to: " + path);
    }
}

Under the Hood: JVM Behavior

VTable & ITable

Khi em gọi một phương thức thông qua Interface, JVM không biết ngay mã máy ở đâu. Nó sử dụng:

  • vtable (Virtual Method Table): Dành cho class inheritance. Mỗi class có một vtable trỏ đến địa chỉ các method thực thi.
  • itable (Interface Method Table): Phức tạp hơn vì một class có thể implement nhiều interface. JVM phải tìm kiếm trong itable để mapping đúng method.
Performance Note: invokeinterface chậm hơn một chút so với invokevirtual (class) do quá trình lookup ITable phức tạp hơn. Tuy nhiên, JIT Compiler hiện đại đã tối ưu hóa điều này cực tốt qua "Inline Caching".

Senior Best Practices

1. Program to Interfaces

Luôn khai báo kiểu là Interface thay vì Concrete Class để linh hoạt chuyển đổi Implementation (Dependency Injection).

2. Interface Segregation (SOLID)

Đừng tạo "Fat Interface". Hãy chia nhỏ interface thành các nhóm chức năng cụ thể.

3. Composition over Inheritance

Dùng abstract class khi các subclass thực sự chia sẻ cấu trúc (is-a). Nếu chỉ muốn mượn tính năng, hãy dùng Interface + Composition.

4. Sealed Classes (Java 17+)

Dùng `sealed` để giới hạn các class nào được phép kế thừa, giúp tăng tính an toàn và tối ưu hóa compiler.

Câu hỏi phỏng vấn hóc búa

Q1: Tại sao Java không cho phép phương thức trừu tượng (abstract) là Static? ▼
Phương thức Static thuộc về class, không phải instance. Abstraction dựa trên Polymorphism (đa hình), nơi phương thức được ghi đè (Override) tại thời điểm thực thi (Runtime) dựa trên đối tượng thực tế. Việc kết hợp static (compile-time) và abstract (runtime override) là mâu thuẫn về mặt logic thiết kế của Java.
Q2: Một Abstract Class có thể có Constructor không? Tại sao? ▼
CÓ. Mặc dù ta không thể khởi tạo trực tiếp (new) một abstract class, nhưng nó vẫn cần constructor để khởi tạo các thuộc tính (fields) mà nó định nghĩa. Khi một subclass được khởi tạo, nó sẽ gọi constructor của cha thông qua super().

Hãy thử tự viết một hệ thống quản lý nhân sự sử dụng cả Interface và Abstract class để nắm vững bài học này nhé!
