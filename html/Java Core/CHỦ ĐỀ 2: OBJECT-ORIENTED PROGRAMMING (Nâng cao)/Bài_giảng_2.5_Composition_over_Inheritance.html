Design Principles

2.5. Composition over Inheritance

Trong lập trình hướng đối tượng, ta thường đứng trước 2 lựa chọn để tái sử dụng mã nguồn:
1. Inheritance (Kế thừa): Lớp con "là một" lớp cha (IS-A).
2. Composition (Thành phần): Lớp mới "có một" đối tượng của lớp khác (HAS-A).

Inheritance (White-box)

Mọi chi tiết của lớp cha đều phơi bày cho lớp con. Rất khó thay đổi lớp cha mà không làm hỏng lớp con.

Composition (Black-box)

Lớp mới chỉ biết "cái gì" lớp thành phần làm được thông qua interface. Dễ dàng thay thế và bảo trì.

Cái bẫy Kế thừa: Fragile Base Class

Hãy xem ví dụ kinh điển khi ta muốn đếm số lượng phần tử được thêm vào một HashSet.

BrokenInheritance.java 
public class InstrumentedHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c); // LỖI Ở ĐÂY!
    }
}
Tại sao nó lỗi? (Self-Recursion)

Bên trong JDK, HashSet.addAll() thực tế lại gọi hàm add() cho từng phần tử.
Nếu em thêm 3 phần tử bằng addAll():
1. addCount tăng lên 3 trong addAll.
2. super.addAll() gọi add() 3 lần.
3. Mỗi lần add() lại nhảy vào hàm đã override của em -> tăng tiếp 3 lần nữa.
Kết quả: addCount = 6 thay vì 3!

Giải pháp: Wrapper Class (Decorator Pattern)

Thay vì kế thừa, ta bao bọc (wrap) thực thể đó và điều hướng (forwarding) các phương thức.

// 1. Lớp chuyển tiếp (Forwarding class)
public class ForwardingSet<E> implements Set<E> {
    private final Set<E> s; // Composition
    public ForwardingSet(Set<E> s) { this.s = s; }

    public boolean add(E e) { return s.add(e); }
    public boolean addAll(Collection<? extends E> c) { return s.addAll(c); }
    // ... implement các methods khác bằng cách gọi s.method()
}

// 2. Lớp Wrapper thực tế
public class InstrumentedSet<E> extends ForwardingSet<E> {
    private int addCount = 0;
    public InstrumentedSet(Set<E> s) { super(s); }

    @Override public boolean add(E e) {
        addCount++;
        return super.add(e);
    }
}
Ưu điểm: Bây giờ InstrumentedSet cực kỳ linh hoạt. Em có thể dùng nó để "đính kèm" tính năng đếm vào HashSet, TreeSet, hay bất kỳ Set nào khác!

Khi nào dùng cái nào?

Dùng Inheritance khi:
  • ✔ Quan hệ thực sự là IS-A (Cat is an Animal).
  • ✔ Lớp cha được thiết kế chuyên biệt để kế thừa (có tài liệu hướng dẫn override).
  • ✔ Cả lớp cha và lớp con nằm trong cùng một package (cùng quyền kiểm soát).
Dùng Composition khi:
  • ✔ Muốn tái sử dụng code từ lớp khác nhưng không muốn bị ràng buộc chặt.
  • ✔ Muốn thay đổi component linh hoạt tại Runtime (Dependency Injection).
  • ✔ Cần tránh "Đa kế thừa" (Java không cho phép nhưng Interface + Composition thì được).

Phỏng vấn Senior

Q: "Inheritance phá vỡ tính đóng gói như thế nào?"

A: Vì lớp con phụ thuộc vào chi tiết triển khai (implementation details) của lớp cha. Nếu lớp cha thay đổi logic bên trong một public method, hoặc thêm một method mới trùng tên với method của lớp con nhưng khác kiểu trả về, lớp con sẽ bị lỗi hoặc chạy sai logic mà không hề biết. Điều này vi phạm nguyên tắc: "Lớp con chỉ nên biết interface của lớp cha, không nên biết lớp cha làm gì bên trong".

Học thiết kế là học cách nhìn xa trông rộng. Hãy dùng Composition nhiều hơn, em nhé!
