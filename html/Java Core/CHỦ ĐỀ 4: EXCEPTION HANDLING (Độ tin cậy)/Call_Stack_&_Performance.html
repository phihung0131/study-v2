Call Stack & Exception Internals

Nhiều người lầm tưởng từ khóa throw hay catch làm chậm chương trình. Sự thật là: **Việc khởi tạo Exception Object** mới là "kẻ sát nhân" bộ nhớ và CPU.

Cơ chế LIFO

Mỗi Thread có một vùng nhớ riêng gọi là **Stack**. Khi một method được gọi, một **Stack Frame** mới được đẩy vào (push). Frame này chứa các biến local và trạng thái thực thi.

main()
processOrder()
validatePayment()
dbQuery()

The "Stack Walk"

Khi Exception xảy ra, JVM phải thực hiện quá trình Stack Walk: Duyệt ngược từ đỉnh Stack về đáy để thu thập thông tin về tên file, tên method và số dòng code.

2. Phương thức fillInStackTrace()

Đây là phương thức Native nằm trong lớp Throwable. Nó được gọi mặc định trong Constructor của mọi Exception.

// Source code từ JDK (Throwable.java)
public synchronized Throwable fillInStackTrace() {
    if (stackTrace != null || backtrace != null /* v.v */) {
        fillInStackTrace(0); // Gọi native method xử lý bên trong JVM
        stackTrace = UNASSIGNED_STACK;
    }
    return this;
}

private native Throwable fillInStackTrace(int dummy);

Cảnh báo hiệu năng: Stack càng sâu (ví dụ trong các framework lớn như Spring hoặc ứng dụng đệ quy), thời gian thực thi của method native này càng lâu.

3. Benchmarks: Thử nghiệm thực tế

Tạo Object thường
~10 ns
Tạo Exception (với Stack)
~5,000 ns
Tạo Fast Exception (không Stack)
~30 ns

* Con số mang tính tương đối, phụ thuộc vào độ sâu của Stack. Exception chậm hơn khoảng 100-500 lần so với Object thường.

4. Java 9+ StackWalker API

Trước Java 9, nếu muốn lấy StackTrace em phải dùng Thread.currentThread().getStackTrace() - nó sẽ snapshot toàn bộ stack ngay lập tức (rất tốn kém). Từ Java 9, StackWalker cho phép chúng ta duyệt Stack một cách Lazy và Stream-based. 

// Chỉ lấy 5 frame đầu tiên - Cực kỳ nhanh!
StackWalker walker = StackWalker.getInstance();
List<String> frames = walker.walk(s -> s.limit(5)
                                       .map(StackFrame::getMethodName)
                                       .collect(Collectors.toList()));

5. OmitStackTraceInFastThrow (Bí ẩn mất log)

💡

Em đã bao giờ thấy log chỉ hiện mỗi java.lang.NullPointerException mà không có dòng code nào chưa?

Đó là do tối ưu hóa của JVM. Khi một exception (như NPE, ArrayIndexOutOfBounds) bị throw liên tục tại cùng một địa điểm code, JIT compiler sẽ nhận ra và chuyển sang chế độ "Fast Throw". Nó sẽ throw một exception đã được tạo sẵn (Pre-allocated) và không có StackTrace để tăng tốc độ.

JVM Flag để tắt tính năng này:
-XX:-OmitStackTraceInFastThrow

6. Thủ thuật tối ưu hóa cho Senior

1. Vô hiệu hóa fillInStackTrace

Dùng cho Business Exceptions dùng để báo hiệu luồng xử lý (Control Flow) mà không cần log stack.

public class FastException extends RuntimeException {
    @Override
    public synchronized Throwable fillInStackTrace() {
        return this; // Không làm gì cả, trả về chính mình
    }
}

2. Sử dụng Singleton Exception (CỰC KỲ CẨN THẬN)

Chỉ dùng khi em không quan tâm đến StackTrace và lỗi này xảy ra cực nhiều.

public class ValidationUtils {
    public static final MyException INVALID_INPUT = new MyException("Invalid Input");
    // Tuy nhiên, StackTrace sẽ luôn chỉ về dòng khai báo static này!
}

7. Interview Questions

Q1: Tại sao việc sử dụng Exception để điều khiển luồng (Control Flow) lại được coi là Anti-pattern?

Đáp án: 1. Hiệu năng tệ (Stack Walk). 2. Làm code khó đọc (Gần giống lệnh GOTO). 3. Khiến trình debug của IDE nhảy lung tung, gây khó khăn cho bảo trì.

Q2: JVM tối ưu hóa "Fast Throw" cho những loại Exception nào?

Đáp án: Chủ yếu là các lỗi nội tại JVM như NPE, ArithmeticException (chia cho 0), ArrayIndexOutOfBoundsException, ArrayStoreException và ClassCastException.
