Vấn đề của cách làm cũ (Try-Finally)

Trước Java 7, để đóng một Resource (File, DB Connection, Socket...), chúng ta phải dùng finally. Cách này có 2 nhược điểm chết người:

1. Boilerplate Code (Rườm rà)

Bạn phải check null, bắt lồng try-catch trong finally. Nếu có 3 resources, code sẽ trở thành một "đống rác" thực sự.

2. Exception Masking (Nuốt lỗi)

Nếu cả try và finally đều ném lỗi, lỗi trong finally sẽ ghi đè lỗi gốc, khiến việc debug trở nên bất khả thi.

// ❌ THE OLD UGLY WAY
InputStream is = null;
try {
    is = new FileInputStream("data.txt");
    // Do something
} catch (IOException e) {
    throw e;
} finally {
    if (is != null) {
        try {
            is.close(); // Nếu dòng này throw exception, exception gốc ở trên sẽ BIẾN MẤT!
        } catch (IOException ex) { /* Swallow or log */ }
    }
}

Hợp đồng AutoCloseable

Để một Class có thể sử dụng trong Try-with-resources, nó bắt buộc phải implement interface java.lang.AutoCloseable.

public interface AutoCloseable {
    /**
     * Được gọi tự động khi kết thúc block try.
     * @throws Exception nếu có lỗi xảy ra khi đóng.
     */
    void close() throws Exception;
}
Lưu ý: Closeable (từ Java 5) kế thừa AutoCloseable nhưng giới hạn throw IOException và yêu cầu tính Idempotent (gọi nhiều lần không sao).

Under the Hood: Compiler làm gì?

Try-with-resources thực chất là một "Syntactic Sugar". Khi biên dịch ra Bytecode, Java Compiler sẽ tự động tạo ra một cấu trúc try-catch-finally cực kỳ phức tạp để đảm bảo an toàn.

Mã bạn viết

try (MyRes res = new MyRes()) {
    res.doWork();
}

Mã sau khi Decompile (Xấp xỉ)

MyRes res = new MyRes();
Throwable primaryExc = null;
try {
    res.doWork();
} catch (Throwable t) {
    primaryExc = t;
    throw t;
} finally {
    if (res != null) {
        if (primaryExc != null) {
            try {
                res.close();
            } catch (Throwable suppressed) {
                // Đây là phép màu!
                primaryExc.addSuppressed(suppressed);
            }
        } else {
            res.close();
        }
    }
}

Suppressed Exceptions

Đây là giải pháp cho vấn đề "nuốt lỗi". Nếu cả doWork() ném lỗi A và close() ném lỗi B:

  • FLOW Lỗi A (Primary) sẽ được ném ra ngoài.
  • FLOW Lỗi B sẽ được "gắn kèm" vào lỗi A dưới dạng Suppressed Exception.

Cách lấy ra:

try {
    // try-with-resources logic
} catch (Exception e) {
    Throwable[] suppressed = e.getSuppressed(); // Lấy mảng các lỗi bị nén lại
}

Cải tiến trong Java 9+

Kể từ Java 9, nếu một biến là effectively final (biến không thay đổi giá trị sau khi gán), bạn có thể đưa trực tiếp nó vào khối try mà không cần khai báo mới.

final Scanner scanner = new Scanner(System.in);
// Java 7, 8: Cần Scanner s2 = scanner; ở đây
try (scanner) { // ✅ Hợp lệ từ Java 9
    // ...
} // scanner sẽ tự động được đóng ở đây

Thử thách Senior

Case 1: Thứ tự đóng Resource

Nếu khai báo try (R1 r1 = new R1(); R2 r2 = new R2()), thì cái nào đóng trước?

Đáp án: Ngược lại với thứ tự khai báo (R2 đóng trước R1). Điều này cực kỳ quan trọng vì R2 có thể phụ thuộc vào R1.

Case 2: "Nuốt lỗi" thủ công

Có cách nào để try-with-resources vẫn "nuốt" mất exception không?

Có: Nếu em catch Exception và throw một Exception hoàn toàn mới mà quên không gán cause hoặc không gọi addSuppressed.
