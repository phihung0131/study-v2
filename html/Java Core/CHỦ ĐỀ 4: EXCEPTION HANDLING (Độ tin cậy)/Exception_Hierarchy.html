Sơ đồ Phân cấp Exception

Trong Java, mọi thứ liên quan đến lỗi đều là Object. Để quản lý hàng nghìn loại lỗi khác nhau, Java thiết lập một hệ thống phân cấp chặt chẽ bắt đầu từ lớp java.lang.Throwable.

Throwable
Error
OutOfMemoryError
StackOverflowError
Exception
Checked Exception
IOException, SQLException
RuntimeException
NullPointerException, ...

1. Error - Những "Cú đấm" chí mạng

Đặc điểm

  • Đại diện cho các vấn đề nghiêm trọng của JVM.
  • Không nên try-catch (vì catch xong cũng không làm gì được).
  • Thường dẫn đến việc dừng chương trình ngay lập tức.

Ví dụ kinh điển

StackOverflowError: Đệ quy vô tận.
OutOfMemoryError: RAM hết, GC bất lực.

2. Checked vs Unchecked Exception

Đây là phần quan trọng nhất khi thiết kế API.

Đặc điểm Checked Exception Unchecked (RuntimeException)
Gốc tọa độ Kế thừa Exception Kế thừa RuntimeException
Thời điểm kiểm tra Compile-time (Bắt buộc xử lý) Runtime (Không bắt buộc)
Ý nghĩa Các sự cố ngoại vi dự đoán được (Network, File) Lỗi logic của lập trình viên (Null, Index out of bound)

3. Internal Working: Chi phí của Exception

Em có biết tại sao tạo một Exception lại tốn kém (Performance hit) không?

Khi một Exception được khởi tạo, JVM phải thực hiện phương thức fillInStackTrace(). Nó sẽ duyệt ngược (walk) toàn bộ call stack hiện tại để ghi lại các frame. Stack càng sâu, Exception càng nặng.

4. Code thực chiến & Anti-patterns

// ❌ ANTI-PATTERN: Swallow Exception (Nuốt lỗi)
try {
    readFile("data.txt");
} catch (IOException e) {
    // Không làm gì cả - Tội ác lớn nhất của Dev!
}

// ✅ BEST PRACTICE: Wrap & Rethrow hoặc Handle triệt để
try {
    readFile("data.txt");
} catch (IOException e) {
    logger.error("Failed to read data file", e);
    throw new BusinessException("Service unavailable", ErrorCode.FILE_NOT_FOUND);
}

Mentor Tip: Đừng bao giờ catch Throwable hay Error trừ khi em đang viết một Framework/Container quản lý vòng đời ứng dụng.
