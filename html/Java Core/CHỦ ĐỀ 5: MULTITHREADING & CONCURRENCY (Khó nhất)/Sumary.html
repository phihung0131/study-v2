OS Level

1. Tiến trình, Luồng & Bài toán CPU Core

Trong Java, việc phân biệt giữa Tiến trình (Process) và Luồng (Thread) là cực kỳ quan trọng để hiểu cách hệ thống quản lý tài nguyên.

Process (Tiến trình)

Một thực thể độc lập được OS cấp phát Heap, Stack riêng. Nếu P1 sập, P2 không ảnh hưởng.

Thread (Luồng)

Đơn vị chạy trong Process. Dùng chung Heap nhưng có Stack riêng. Một lỗi nặng ở Thread có thể làm sập cả Process.

Câu hỏi: Máy 4 Core, chạy 8 Thread thì sao?
Trả lời: OS dùng CPU Scheduling. Tại một tích tắc (ms), 1 Core chỉ chạy 1 Thread. Nó hoán đổi cực nhanh (Context Switch) khiến ta cảm giác chúng chạy song song. Nếu có Hyper-Threading, 1 Core vật lý sẽ có 2 Core logic, giúp xử lý 8 luồng mượt mà hơn.

2. Thread Lifecycle (Vòng đời luồng)

JVM quản lý luồng qua 6 trạng thái chính. Hiểu điều này giúp debug "treo" ứng dụng:

  • NEW: Vừa khởi tạo đối tượng Thread, chưa gọi start().
  • RUNNABLE: Đã sẵn sàng, đang chiếm CPU hoặc đang đợi trong hàng đợi của OS.
  • BLOCKED: Đứng đợi cửa để lấy khóa synchronized.
  • WAITING: Đợi vô thời hạn (gọi wait(), join()) cho đến khi ai đó đánh thức.
  • TIMED_WAITING: Đợi có thời hạn (gọi sleep(ms)).
  • TERMINATED: Kết thúc công việc hoặc bị lỗi.

3. Khởi tạo & Native Trace-off

Khi gọi start(), JVM gọi xuống hàm native start0() của C++. OS sẽ tạo một luồng thật và cấp phát 1MB Stack.

// Cách khuyên dùng: Implement Runnable để tách biệt Logic và Worker
Thread t = new Thread(() -> {
    System.out.println("Task đang chạy trên: " + Thread.currentThread().getName());
});
t.start();
Trade-off: Việc tạo 1 Thread rất đắt đỏ. Nếu bạn nhận 1000 request/s và new Thread() cho mỗi cái, bạn sẽ tốn 1GB RAM chỉ để giữ luồng và CPU sẽ chết vì Context Switching (thay vì làm việc).

4. Thread Pool & Hiện tượng Bottleneck

Thread Pool giống như một đội công nhân cố định để tránh chi phí tuyển dụng (tạo luồng) liên tục.

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, // Core size
    10, // Max size
    60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100), // Điểm nghẽn ở đây nếu hàng đợi đầy
    new ThreadPoolExecutor.CallerRunsPolicy()
);
Tại sao Core có 4 mà Pool có 50?
Vì phần lớn thời gian luồng là I/O Bound (đợi Database, đợi Mạng). Khi 46 ông đang "ngủ" đợi dữ liệu, 4 ông còn lại vẫn có thể dùng CPU để xử lý.

5. Race Condition: Bài toán Rút tiền ATM

Đây là lỗi khi 2 luồng cùng truy cập và sửa đổi dữ liệu chung dẫn đến kết quả sai lệch.

public synchronized void withdraw(int amount) {
    if (balance >= amount) {
        // Giả sử Luồng 1 bị dừng ở đây sau khi check OK
        Thread.sleep(100); 
        balance -= amount;
    }
}
Sử dụng synchronized sẽ khóa đối tượng lại, chỉ cho 1 người vào thực hiện giao dịch tại 1 thời điểm.
Under the Hood

6. Intrinsic Locks & Monitor

Mỗi Object trong Java có một Mark Word chứa thông tin khóa. synchronized sử dụng cơ chế này.

  • Entry Set: Nơi các luồng đứng đợi lấy khóa.
  • Wait Set: Nơi các luồng tạm buông khóa (gọi wait()) để chờ điều kiện.
  • Reentrant: Nếu bạn đã giữ khóa, bạn có thể gọi các hàm synchronized khác của cùng đối tượng đó mà không bị kẹt.
Optimization

7. Volatile vs Atomic: Visibility & Atomicity

Volatile

Giải quyết vấn đề Visibility. Đảm bảo luồng đọc luôn thấy giá trị mới nhất từ RAM, không đọc từ CPU Cache.

AtomicInteger

Sử dụng CAS (Compare-And-Swap) ở tầng CPU. Không cần Lock nhưng vẫn đảm bảo phép toán count++ an toàn.

8. Explicit Locks (ReentrantLock)

Mạnh hơn synchronized vì có khả năng "rút lui" và "thử khóa".

if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try {
        // Xử lý an toàn
    } finally {
        lock.unlock();
    }
} else {
    System.out.println("Hệ thống bận, vui lòng thử lại sau");
}

Ví dụ Kho hàng: Luồng "Shipper" phải await() nếu kho đầy. Luồng "Khách hàng" sau khi mua sẽ signal() cho Shipper biết kho đã trống chỗ.

10. Java Concurrency Utility (JUC)

CountDownLatch: Chờ N sự kiện xong mới chạy tiếp (ví dụ: Chờ 3 Service Database, Cache, Mail khởi động xong).
Semaphore: Giới hạn số lượng luồng vào một tài nguyên (ví dụ: tối đa 10 kết nối Database).
ConcurrentHashMap: Khóa theo từng Bucket thay vì khóa cả Map, giúp hàng nghìn luồng đọc ghi song song.

11. Fork/Join & CompletableFuture

Fork/Join: Đội quân kiến

Chia nhỏ 10 triệu giao dịch thành các phần 1000 và tính tổng song song bằng Work-Stealing.

CompletableFuture: Dây chuyền tự động

Lập trình bất đồng bộ (Non-blocking). Phù hợp gọi API: Gọi hãng bay + Gọi ngân hàng -> Có kết quả thì Gửi Email.

12. Virtual Threads (Project Loom)

Cuộc cách mạng của Java 21. Cho phép tạo 1.000.000 luồng trên một máy tính cá nhân.

Cơ chế Mount/Unmount

Khi luồng ảo đợi mạng, JVM bốc nó ra khỏi CPU, lưu trạng thái vào Heap. Khi có mạng, JVM nạp lại và chạy tiếp.

Lợi ích

Viết code dễ hiểu kiểu tuần tự nhưng hiệu năng cực cao như lập trình Reactive phức tạp.

Enterprise

13. Distributed Consistency (Spring Boot)

Khi chạy nhiều Server, khóa trong Java vô tác dụng. Chúng ta dùng Database làm trọng tài.

// Optimistic Locking với JPA
@Entity
public class Account {
    private Double balance;
    @Version
    private Integer version; // DB tự kiểm tra version khi update
}

Distributed Lock với Redis: Dùng để "xếp hàng" người dùng ngay từ tầng ngoài cùng cho các sự kiện như Flash Sale.

Đa luồng không làm CPU mạnh hơn, nó chỉ giúp CPU không bị rảnh rỗi.
