Chapter 5.3

Volatile & Atomic:
Cơ chế Non-blocking trong Java

Tại sao đôi khi synchronized lại là "overkill"? Làm sao để các Thread giao tiếp với nhau mà không cần chờ đợi? Hãy cùng khám phá tầng sâu của Java Memory Model (JMM).

01. Volatile: Phá vỡ rào cản Cache

Bài toán Visibility: Mỗi CPU Core có một vùng L1/L2 Cache riêng. Khi Thread 1 sửa biến running = false, giá trị này có thể chỉ nằm ở Cache của Core 1. Thread 2 (chạy ở Core 2) vẫn thấy running = true và lặp vô tận.

Core 1 (Thread 1)
true
Core 2 (Thread 2)
true
Main Memory (RAM): true

Từ khóa 'volatile' làm được gì?

  • ✔ Visibility: Mọi thao tác ghi vào biến volatile sẽ được flush xuống Main Memory ngay lập tức. Mọi thao tác đọc sẽ đọc trực tiếp từ Main Memory.
  • ✔ No Reordering: Ngăn chặn trình biên dịch và CPU sắp xếp lại thứ tự các lệnh (Instruction Reordering).

02. Cảnh báo: Volatile không phải là Atomic

Sai lầm kinh điển của Senior!

Nhiều người nghĩ public volatile int count = 0; và thực hiện count++ là an toàn. **KHÔNG!**

Thao tác count++ (3 bước):

  1. Đọc count (đảm bảo visibility).
  2. Tăng count + 1.
  3. Ghi lại count.

Nếu 2 thread cùng ở bước 2, chúng sẽ cùng ghi một giá trị cũ ở bước 3!

// SAI TRONG ĐA LUỒNG
volatile int count = 0;

public void inc() {
    count++; // Vẫn bị Race Condition!
}

03. Atomic Classes: Giải pháp Non-blocking

Gói java.util.concurrent.atomic cung cấp các class như AtomicInteger, AtomicLong, AtomicReference... Chúng đảm bảo các thao tác là **Atomic** mà không cần dùng synchronized.

import java.util.concurrent.atomic.AtomicInteger;

public class Counter {
    private AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // Hoàn toàn atomic!
    }
}

Deep Dive: Cơ chế CAS (Compare-And-Swap)

Bí mật đằng sau Atomic Classes không phải là Lock, mà là lệnh CPU **CAS**. Nó hoạt động theo nguyên tắc: "Tôi nghĩ giá trị hiện tại là V, nếu đúng là V thì hãy đổi nó thành N. Nếu không phải V (nghĩa là ai đó đã sửa rồi), tôi sẽ thử lại từ đầu."

Đọc giá trị hiện tại (Expected: 5)

CPU CAS Command

Compare(current, expected)

Bằng nhau? -> Ghi giá trị mới (6)
Khác nhau? -> Loop lại từ đầu (Retry)

Đây là lý do tại sao nó được gọi là **Optimistic Locking** (Khóa lạc quan).

So sánh: Khi nào dùng gì?

Công cụ Đặc tính Chi phí (Overhead) Tình huống tốt nhất
synchronized Blocking, Lock-based Cao (Context Switch) Logic phức tạp, nhiều biến liên quan
volatile Visibility Only Rất thấp Flag dừng thread (stop flag)
AtomicInteger CAS, Lock-free Trung bình (Retry loop) Counter, ID generator, tranh chấp nhẹ

Thực hành cùng Mentor

Đề bài:

Bạn đang xây dựng một hệ thống giới hạn lượt truy cập (Rate Limiter). Bạn cần một biến count để đếm số request trong 1 giây. Có 100 threads cùng gọi hàm này.

Câu hỏi:

1. Nếu dùng volatile int, kết quả có chính xác không?

2. Nếu dùng AtomicInteger, khi nào hiệu năng của nó sẽ tệ hơn synchronized?

Mentor Gợi ý: Khi sự tranh chấp (contention) cực kỳ cao, hàng ngàn thread cùng CAS vào 1 địa chỉ, vòng lặp 'retry' của Atomic sẽ ngốn rất nhiều CPU. Lúc này, LongAdder (một bản nâng cao của Atomic trong Java 8) sẽ là cứu cánh!
