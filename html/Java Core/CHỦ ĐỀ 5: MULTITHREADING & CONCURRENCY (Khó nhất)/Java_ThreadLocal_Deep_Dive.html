ThreadLocal:
Dữ liệu Riêng tư tuyệt đối

Trong khi synchronized cố gắng điều phối nhiều thread truy cập 1 biến, thì ThreadLocal giải quyết vấn đề bằng cách cung cấp cho mỗi thread một bản sao riêng của biến đó.

Mô phỏng: "Vách ngăn" giữa các luồng

Mentor Note: Cả hai Thread cùng gọi biến context, nhưng kết quả nhận được là hoàn toàn độc lập. Không có sự tranh chấp, không cần Lock, hiệu năng đạt mức tối đa.

Triển khai ThreadLocal

UserContextHolder.java Production Style 
public class UserContextHolder {
    // Khởi tạo với giá trị mặc định
    private static final ThreadLocal<User> userContext = 
        ThreadLocal.withInitial(() -> null);

    public static void setUser(User user) {
        userContext.set(user);
    }

    public static User getUser() {
        return userContext.get();
    }

    public static void clear() {
        userContext.remove(); // Cực kỳ quan trọng!
    }
}

Ứng dụng phổ biến:

  • • Lưu trữ Transaction ID hoặc User Session xuyên suốt các tầng (Controller -> Service -> Repo).
  • • Xử lý các đối tượng không an toàn đa luồng (ví dụ: SimpleDateFormat).
  • • Database Connection Management (mỗi thread một kết nối).

Internal Working: ThreadLocalMap

Sự thật bất ngờ: Dữ liệu không được lưu bên trong đối tượng ThreadLocal.

Nó được lưu ngay bên trong mỗi đối tượng Thread dưới dạng một biến instance gọi là ThreadLocalMap.

Cấu trúc Entry:

Entry extends WeakReference<ThreadLocal<?>>

Key là một WeakReference tới ThreadLocal object. Value là giá trị bạn set.

Thread Object
⬇ contains ⬇

ThreadLocalMap

[Key: ThreadLocal, Value: Object]

Cảnh báo: Memory Leak trong Thread Pool

Tại sao Leak xảy ra?

Trong môi trường Web Server (Tomcat/Spring), các Thread được tái sử dụng qua **Thread Pool**.

Nếu bạn gọi set() mà quên gọi remove(), dữ liệu sẽ ở lại mãi mãi trong ThreadLocalMap của Thread đó, kể cả khi Request đã kết thúc.

Hậu quả: 1. **Sai lệch dữ liệu:** Request sau có thể nhìn thấy User của Request trước.
2. **Tràn bộ nhớ:** Các Object lớn không bao giờ được GC giải phóng.

Giải pháp Senior:

// Luôn sử dụng try-finally
try {
    UserContextHolder.setUser(currentUser);
    processRequest();
} finally {
    UserContextHolder.clear(); // BẮT BUỘC
}

ThreadLocal trong kỷ nguyên Java 21

Với **Virtual Threads**, bạn có thể có hàng triệu Thread. Nếu mỗi Thread tạo một ThreadLocalMap lớn, RAM của bạn sẽ bốc hơi ngay lập tức.

Rủi ro

InheritableThreadLocal sẽ rất đắt đỏ khi copy dữ liệu qua hàng triệu Virtual Thread con.

Tương lai: Scoped Values

Java 21 giới thiệu ScopedValue như một giải pháp thay thế nhẹ nhàng hơn, Immutable và an toàn hơn cho Virtual Threads.

Câu hỏi Mentor (Interview Prep)

Q: Tại sao Key trong ThreadLocalMap lại là WeakReference?

A: Để khi ThreadLocal object không còn biến nào tham chiếu tới nữa, Garbage Collector có thể thu hồi nó. Nếu là Strong Reference, Key này sẽ giữ cho toàn bộ Entry sống mãi cùng Thread (Memory Leak).

Q: Nếu dùng WeakReference rồi, tại sao vẫn phải gọi remove()?

A: Vì WeakReference chỉ áp dụng cho Key. Phần Value vẫn là Strong Reference. Nếu không gọi remove(), Value sẽ bị treo lơ lửng cho đến khi Thread chết.
