MODULE 5.2

Synchronization:
Thiết lập Trật tự trong Đa luồng

Khi nhiều Thread cùng truy cập và sửa đổi dữ liệu dùng chung, chúng ta đối mặt với thảm họa dữ liệu. synchronized là cơ chế "loại trừ tương hỗ" (Mutual Exclusion) giúp đảm bảo tại một thời điểm chỉ có một Thread được phép thực thi đoạn code nhạy cảm.

01 Vấn đề: Race Condition

Xét ví dụ kinh điển: Hai Thread cùng tăng một biến count++ 10,000 lần. Kết quả mong đợi là 20,000 nhưng thực tế thường thấp hơn. Tại sao?

Cơ chế của count++:

  1. Đọc giá trị từ Memory vào Register.
  2. Tăng giá trị trong Register.
  3. Ghi ngược lại vào Memory.

-> Nếu Thread A đang ở bước 2 mà Thread B nhảy vào bước 1, giá trị của A sẽ bị đè mất (Lost Update).

// Unsafe code
public void increment() {
    count++; // Race condition ở đây!
}

02 Cú pháp Synchronized

Instance Methods

public synchronized void method() {
    // Khóa trên đối tượng 'this'
}

Dùng khi muốn bảo vệ toàn bộ instance.

Static Methods

public static synchronized void log() {
    // Khóa trên Class object (.class)
}

Dùng cho các tài nguyên static chung toàn hệ thống.

Synchronized Block (Khuyên dùng)

private final Object lock = new Object();
public void update() {
    synchronized(lock) {
        // Chỉ code bên trong mới được bảo vệ
    }
}

Mentor Tip: Luôn dùng block với một Object lock riêng để tránh "Lock Contention" không đáng có trên toàn bộ instance.

03 Bản chất: Monitor Lock (Intrinsic Lock)

Trong JVM, mọi Object đều có một Header chứa thông tin gọi là Mark Word. Khi một Thread thực hiện synchronized, nó sẽ cố gắng ghi ID của mình vào Mark Word này.

  • Biased Locking: Nếu chỉ 1 thread dùng lock, JVM sẽ "thiên vị" cho nó để giảm chi phí.
  • Lightweight Locking: Khi có 2 thread bắt đầu tranh chấp nhẹ, JVM dùng CAS (Compare-And-Swap).
  • Heavyweight Locking: Khi tranh chấp cao, Thread bị đẩy vào Entry Set và chuyển sang trạng thái BLOCKED.

Hai "Phép màu" của Synchronized

1. Atomicity (Tính nguyên tử)

Đảm bảo một nhóm các câu lệnh được thực thi như một đơn vị duy nhất, không thể bị chia cắt bởi Thread khác.

2. Visibility (Tính hiển thị)

Quan trọng! Khi Thread thoát khỏi synchronized block, nó bắt buộc phải đẩy (flush) mọi thay đổi từ Local Cache vào Main Memory, giúp các Thread khác thấy được kết quả mới nhất.

Mentor's Best Practices

!

Giữ vùng găng (Critical Section) nhỏ nhất có thể.

Chỉ bao bọc code thực sự cần bảo vệ để tránh giảm hiệu năng hệ thống.

!

Ưu tiên 'private final Object lock'.

Tránh dùng String literal làm lock vì có thể bị trùng trong String Pool, gây deadlock xuyên ứng dụng.

!

Synchronized là Reentrant (Có thể vào lại).

Một Thread đang giữ Lock có thể gọi tiếp một method synchronized khác của chính nó mà không bị kẹt.

Câu hỏi tư duy (Senior Level)

"Tại sao đôi khi dùng synchronized vẫn thấy ứng dụng bị chậm đi đáng kể?"

Đó là hiện tượng Lock Contention. Khi quá nhiều Thread đứng xếp hàng đợi một cái Lock duy nhất, thời gian chờ đợi lớn hơn thời gian xử lý thực tế. Lúc này bạn nên cân nhắc chia nhỏ Lock (Lock Stripping) hoặc dùng các cấu trúc Non-blocking như AtomicInteger.
