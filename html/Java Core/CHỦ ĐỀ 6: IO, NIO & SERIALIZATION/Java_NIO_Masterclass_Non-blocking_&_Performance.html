Zero-Copy: Tối thượng của truyền tải

Trong I/O truyền thống, để gửi một file qua Network, dữ liệu phải được copy qua lại 4 lần giữa Kernel Space và User Space (JVM). Zero-copy loại bỏ các bước trung gian thừa thãi này.

APP / JVM (User Space)
KERNEL (OS Space)
DISK / NIC (Hardware)

Senior Code: transferTo()


// Sử dụng FileChannel để thực hiện Zero-copy
try (FileChannel source = new FileInputStream("large_video.mp4").getChannel();
     SocketChannel target = SocketChannel.open(new InetSocketAddress("remote-host", 8080))) {
    
    // Dữ liệu đi thẳng từ Disk Cache sang Network Buffer ở tầng OS
    source.transferTo(0, source.size(), target);
}
Tại sao nhanh? Vì nó giảm thiểu "Context Switching" giữa User mode và Kernel mode, đồng thời không chiếm dụng CPU để copy dữ liệu vô nghĩa. Kafka dùng chiêu này để đạt tốc độ hàng triệu message/s.

Serialization Deep Dive

Serialization là quá trình biến một Object graph thành một dãy các bytes để lưu trữ hoặc truyền đi. Nhưng cẩn thận, nó được coi là "con ngựa thành Troy" của Java.

Các quy tắc "Sống còn"

  • 🔹 Phải implement Serializable.
  • 🔹 serialVersionUID: Luôn luôn phải khai báo thủ công để tránh lỗi InvalidClassException khi class thay đổi nhẹ.
  • 🔹 transient: Dùng cho các field nhạy cảm (password) hoặc không thể serialize (Thread, Connection).

Mối nguy bảo mật

Hàm readObject() bản chất là một "Ẩn số Constructor". Kẻ tấn công có thể chế tạo một chuỗi byte độc hại để khi Java deserialize, nó sẽ thực thi mã độc hoặc tạo ra các Object vi phạm tính đóng gói. 


public class User implements Serializable {
    private static final long serialVersionUID = 1L; // Bắt buộc!
    
    private String username;
    private transient String password; // Sẽ không bị lưu lại
    
    // Customize quá trình Serialization (Hiếm khi dùng nhưng Senior phải biết)
    private void writeObject(ObjectOutputStream out) throws IOException {
        out.defaultWriteObject(); // Ghi mặc định
        // Thêm logic mã hóa thủ công ở đây nếu cần
    }
}

Serialization Proxy Pattern

Joshua Bloch khuyên dùng pattern này để triệt tiêu mọi lỗ hổng bảo mật của Serialization truyền thống.

Tư duy Proxy:

Thay vì Serialize chính Object đó, ta Serialize một "đại diện" (Proxy) cực kỳ đơn giản của nó. Khi Deserialize, Proxy sẽ tự gọi Constructor chuẩn của Object thật để khởi tạo lại.


public final class Period implements Serializable {
    private final Date start;
    private final Date end;

    // Inner class làm Proxy
    private static class SerializationProxy implements Serializable {
        private final Date start;
        private final Date end;
        SerializationProxy(Period p) { this.start = p.start; this.end = p.end; }
        
        // Sau khi deserialize Proxy, nó sẽ "hóa thân" thành Period thật
        private Object readResolve() {
            return new Period(start, end); // Gọi constructor chuẩn -> Invariants được check!
        }
    }

    // Khi được yêu cầu serialize, thay thế Period bằng Proxy
    private Object writeReplace() {
        return new SerializationProxy(this);
    }
}
