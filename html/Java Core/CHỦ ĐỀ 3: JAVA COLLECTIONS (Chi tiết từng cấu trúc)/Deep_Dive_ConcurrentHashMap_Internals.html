ConcurrentHashMap

Khám phá bí mật đằng sau cấu trúc dữ liệu Thread-safe nhanh nhất thế giới Java, nơi Locking không còn là "nỗi sợ" của Performance.

1 Vấn đề của SynchronizedMap / Hashtable

The "Giant Lock" Problem

`Collections.synchronizedMap` hoặc `Hashtable` sử dụng một Monitor Lock duy nhất trên **toàn bộ đối tượng Map**.

Khi Luồng A đang `put`, Luồng B muốn `get` (ở một bucket hoàn toàn khác) cũng phải đứng chờ. Điều này tạo ra "nút thắt cổ chai" (bottleneck) kinh khủng trong hệ thống High-Concurrency.

Mô phỏng Luồng

Thread A: Writing... [LOCKING MAP]
Thread B: Waiting to Read...
Thread C: Waiting to Put...
Global Lock = Low Throughput

2 Sự tiến hóa của ConcurrentHashMap

Java 7: Segment Locking (Lock Striping)

Map được chia thành 16 phân đoạn (Segments). Mỗi Segment là một con Map nhỏ có Lock riêng. Luồng A thao tác trên Segment 1 không làm ảnh hưởng Luồng B trên Segment 2.
Nhược điểm: Vẫn tốn bộ nhớ cho các Segment và số lượng song song bị giới hạn bởi số Segment.

Java 8+: CAS & Synchronized on Bucket Head

Bỏ hoàn toàn Segments. Lock được đặt trực tiếp lên **Node đầu tiên của mỗi Bucket**. Nếu Bucket trống, Java dùng **CAS (Compare-And-Swap)** để chèn Node mà không cần dùng Lock (Lock-free). Nếu Bucket đã có dữ liệu, chỉ đồng bộ hóa (synchronized) trên Node đầu tiên đó.

3 CAS (Compare-And-Swap) & volatile

`ConcurrentHashMap` sử dụng các phép toán nguyên tử cấp độ CPU. Thay vì nói "Hãy khóa mảng lại cho tôi", nó nói: "Nếu giá trị tại ô này hiện tại là A, hãy đổi nó thành B". Nếu đúng là A, việc đổi xảy ra tức thì. Nếu không, luồng sẽ thử lại (Retry).

// Pseudo-code logic của put() trong Java 8
for (Node[] tab = table;;) {
  if (bucketIsEmpty(i)) {
    if (casTabAt(i, null, newNode)) break; // Không dùng lock!
  } else {
    synchronized (nodeAt(i)) { // Chỉ lock bucket này
      updateNode();
    }
  }
}

Từ khóa `volatile`

Mảng `table` và các biến `val`, `next` trong Node được đánh dấu là `volatile`. Điều này đảm bảo tính **Visibility** (Tầm nhìn): Một luồng vừa sửa dữ liệu, các luồng khác sẽ thấy ngay lập tức ở mức độ RAM, không bị cache ở CPU Register.

5 Weakly Consistent Iterators

Tạm biệt ConcurrentModificationException!

Không giống `ArrayList` hay `HashMap`, Iterator của `ConcurrentHashMap` **không ném lỗi** khi Map bị sửa đổi trong lúc duyệt. Nó phản ánh trạng thái của Map tại thời điểm Iterator được tạo và CÓ THỂ (nhưng không hứa trước) thấy các thay đổi sau đó. Đây gọi là tính nhất quán yếu (Weakly Consistent).

6 Atomic Operations: Đừng dùng if-else!

Trong môi trường đa luồng, việc kiểm tra rồi mới làm (Check-then-act) là cực kỳ nguy hiểm. Hãy sử dụng các phương thức nguyên tử:

putIfAbsent(K, V)

Chỉ thêm nếu Key chưa tồn tại. Trả về giá trị cũ.

computeIfAbsent(...)

Tính toán và thêm giá trị mới chỉ khi Key vắng mặt.

merge(K, V, Func)

Gộp giá trị cũ và mới theo một logic định sẵn.

Thử thách Mentor (Interview Prep)

Q1: ConcurrentHashMap có cho phép null key/value không?

Mentor Answer: KHÔNG. Cả Key và Value đều không được null. Lý do là để tránh sự nhập nhằng trong môi trường đa luồng (ví dụ: nếu `get(key)` trả về `null`, ta không biết là Key không tồn tại hay Value thực sự là `null`).

Q2: size() của ConcurrentHashMap có chính xác tuyệt đối không?

Mentor Answer: Nó là một giá trị ước tính tốt nhất. Vì tại thời điểm hàm `size()` đang đếm, các luồng khác có thể đang thêm/xóa. Tuy nhiên, nó nhanh hơn rất nhiều so với việc khóa toàn bộ Map để đếm.

Q3: Khi nào nên dùng Collections.synchronizedMap thay vì ConcurrentHashMap?

Mentor Answer: Rất hiếm khi. `synchronizedMap` chỉ nên dùng khi bạn cần khóa toàn bộ Map để thực hiện một chuỗi các thao tác nguyên tử thủ công (ví dụ: duyệt qua toàn bộ Map và thay đổi dữ liệu theo một transaction).
