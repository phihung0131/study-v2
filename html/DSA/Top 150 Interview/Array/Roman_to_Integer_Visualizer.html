Roman to Integer

LeetCode #13 | Difficulty: Easy | Math, String, Hash Table

📖 Problem Summary

Roman numerals are represented by seven different symbols: I, V, X, L, C, D, M. Usually, they are written largest to smallest from left to right. However, there are six subtraction cases:

I before V/X (4, 9)
X before L/C (40, 90)
C before D/M (400, 900)

Goal: Convert a given Roman string (e.g., "MCMXCIV") into its integer value (1994).

APPROACH 1 | STRING PRE-PROCESSING

Intuition: "The Simplifier"

The hardest part is the subtraction logic. Why not eliminate it? We can replace "IV" with "IIII", "IX" with "VIIII", etc. Once all subtractive pairs are replaced, every symbol in the string is additive, making the sum trivial.

Java Implementation

public int romanToInt(String romanString) {
    // Pre-process the string to remove subtraction complexity
    romanString = romanString.replace("IV", "IIII")
                             .replace("IX", "VIIII")
                             .replace("XL", "XXXX")
                             .replace("XC", "LXXXX")
                             .replace("CD", "CCCC")
                             .replace("CM", "DCCCC");

    int totalIntegerSum = 0;
    for (char symbol : romanString.toCharArray()) {
        if (symbol == 'M') totalIntegerSum += 1000;
        else if (symbol == 'D') totalIntegerSum += 500;
        else if (symbol == 'C') totalIntegerSum += 100;
        else if (symbol == 'L') totalIntegerSum += 50;
        else if (symbol == 'X') totalIntegerSum += 10;
        else if (symbol == 'V') totalIntegerSum += 5;
        else if (symbol == 'I') totalIntegerSum += 1;
    }
    return totalIntegerSum;
}

Complexity

Time: O(N). Replacement and iteration are linear.

Space: O(N). Creating a new, much longer string.

Pros & Cons

Pros: Logic is simple and error-proof.

Cons: Very slow due to multiple string allocations and regex searches.

APPROACH 2 | MAP LOOKUP & LOOKAHEAD

Intuition: "The Forward Scan"

Iterate from left to right. For each symbol, check the next one. If the current value is smaller than the next value (e.g., I < V), we subtract the current value. Otherwise, we add it. This handles subtraction naturally without modifying the string.

Java Implementation

public int romanToInt(String s) {
    Map<Character, Integer> symbolValues = new HashMap<>();
    symbolValues.put('I', 1); symbolValues.put('V', 5);
    symbolValues.put('X', 10); symbolValues.put('L', 50);
    symbolValues.put('C', 100); symbolValues.put('D', 500);
    symbolValues.put('M', 1000);

    int totalIntegerResult = 0;
    for (int i = 0; i < s.length(); i++) {
        int currentValue = symbolValues.get(s.charAt(i));
        
        // Check lookahead to see if we should subtract
        if (i + 1 < s.length() && currentValue < symbolValues.get(s.charAt(i + 1))) {
            totalIntegerResult -= currentValue;
        } else {
            totalIntegerResult += currentValue;
        }
    }
    return totalIntegerResult;
}

Complexity

Time: O(N). Single pass through the string.

Space: O(1). Map size is constant (7 symbols).

Standard Pattern

This is the most common interview approach. It demonstrates clear understanding of string iteration and dictionary usage.

APPROACH 3 | REVERSE SCAN & SWITCH

Intuition: "The Backwards Hack"

If we iterate backwards, we don't need a map or lookahead logic. We just keep track of the maximum value seen so far. If current value is less than max seen, subtract it. Else, add it and update max. Using a switch statement instead of a Map is much faster in Java.

Java Implementation (Optimal)

public int romanToInt(String s) {
    int totalSum = 0;
    int maxValueSeen = 0;
    
    // Iterate backwards - simplifies the logic significantly
    for (int i = s.length() - 1; i >= 0; i--) {
        int currentNumValue = switch (s.charAt(i)) {
            case 'M' -> 1000;
            case 'D' -> 500;
            case 'C' -> 100;
            case 'L' -> 50;
            case 'X' -> 10;
            case 'V' -> 5;
            case 'I' -> 1;
            default -> 0;
        };
        
        // Comparison determines addition or subtraction
        if (currentNumValue < maxValueSeen) {
            totalSum -= currentNumValue;
        } else {
            totalSum += currentNumValue;
            maxValueSeen = currentNumValue;
        }
    }
    return totalSum;
}

Efficiency

Time: O(N). Best case (no map overhead).

Space: O(1). Constant space.

Why Switch?

Java's HashMap involves object creation, hashing, and bucket collisions. A switch compiles to lookupswitch or tableswitch bytecode, which is basically an O(1) jump in machine code.

⚡ Fast Recall Cheat Sheet

Core Patterns

When converting Roman numerals, you have two logical choices:

  • 1. Forward Scan: Compare current < next. If true, subtract current.
  • 2. Backward Scan: Track maxSeen. If current < maxSeen, subtract current.
