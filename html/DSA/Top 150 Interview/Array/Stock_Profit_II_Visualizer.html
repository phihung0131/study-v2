Best Time to Buy and Sell Stock II

LeetCode #122 | Difficulty: Medium | Array, Greedy, DP

📈 Problem Summary

You are given an integer array prices. On each day, you may decide to buy and/or sell the stock. You can only hold at most one share of the stock at any time. However, you can buy it then immediately sell it on the same day.

Goal: Find the maximum profit you can achieve by making as many transactions as you like.

Approach 1: Recursive Backtracking (Brute Force)

Intuition

In a brute-force approach, we calculate the profit of every possible combination of transactions. For each day, we can choose to either buy or skip. If we buy, we then explore all future days to sell. This creates a massive recursion tree.

Java Implementation

public int maxProfit(int[] prices) {
    return calculate(prices, 0);
}

private int calculate(int[] prices, int s) {
    if (s >= prices.length) return 0;
    int max = 0;
    for (int start = s; start < prices.length; start++) {
        int maxProfit = 0;
        for (int i = start + 1; i < prices.length; i++) {
            if (prices[start] < prices[i]) {
                int profit = calculate(prices, i + 1) + prices[i] - prices[start];
                if (profit > maxProfit) maxProfit = profit;
            }
        }
        if (maxProfit > max) max = maxProfit;
    }
    return max;
}

Complexity Analysis

Time Complexity: O(nⁿ). Recursive calls grow exponentially.

Space Complexity: O(n). Depth of recursion stack.

Problem

The same subproblems are solved repeatedly. For an array of size 30, this would take billions of operations.

Approach 2: Peak Valley (Graphical)

Intuition

Imagine the price chart as mountains and valleys. To maximize profit, we want to buy at every Valley (local minimum) and sell at the immediately following Peak (local maximum). Summing these segments gives the total max profit.

Java Implementation

public int maxProfit(int[] prices) {
    int i = 0;
    int valley = prices[0];
    int peak = prices[0];
    int totalProfit = 0;
    
    while (i < prices.length - 1) {
        // 1. Find the Valley (dip)
        while (i < prices.length - 1 && prices[i] >= prices[i + 1]) i++;
        valley = prices[i];
        
        // 2. Find the Peak (top)
        while (i < prices.length - 1 && prices[i] <= prices[i + 1]) i++;
        peak = prices[i];
        
        totalProfit += peak - valley;
    }
    return totalProfit;
}

Complexity Analysis

Time Complexity: O(n). Single pass through the array.

Space Complexity: O(1). Constant extra space.

Logic

This approach mirrors the way a human visually identifies growth periods on a stock chart.

Approach 3: Simple Pass (Greedy)

Intuition: "The Daily Gain"

The Peak-Valley approach can be simplified further. If we look at any continuous growth (Valley to Peak), the total profit is exactly the same as the sum of every daily increase.

Example: Buy at 1, Sell at 5 (Profit: 4). If prices were [1, 2, 5], daily gains are (2-1)=1 and (5-2)=3. Total: 1 + 3 = 4. We just add up every prices[i] - prices[i-1] if it is positive!

Java Implementation (Optimal)

public int maxProfit(int[] prices) {
    int maxProfit = 0;
    
    // Iterate from the second day
    for (int i = 1; i < prices.length; i++) {
        // If today's price is higher than yesterday...
        if (prices[i] > prices[i - 1]) {
            // ...add the difference to total profit
            maxProfit += prices[i] - prices[i - 1];
        }
    }
    
    return maxProfit;
}

Complexity Analysis

Time Complexity: O(n). Single loop pass.

Space Complexity: O(1). Only one variable to store profit.

The "Aha!" Moment

"If tomorrow is better than today, I want that money." This Greedy approach effectively captures every single climbing segment of the stock chart.

Efficiency Comparison

Approach Time Space Verdict
Recursive Brute Force O(nⁿ) O(n) Unusable
Peak Valley O(n) O(1) Excellent
Simple Pass (Greedy) O(n) O(1) Optimal / Simplest

⚡ Cheat Sheet

  • • Rule: If prices[i] > prices[i-1], add profit.
  • • Pattern: Greedy accumulation.
  • • Constraint: No overlapping transactions (already handled by greedy logic because buying at $i-1$ and selling at $i$ then buying at $i$ and selling at $i+1$ is equivalent to buying at $i-1$ and selling at $i+1$).

🧩 Common Confusion

Users often ask: "Can I sell on Day 3 and buy again on Day 3?" Yes. In the code, adding prices[3]-prices[2] and prices[4]-prices[3] is mathematically identical to buying on Day 2 and selling on Day 4. The greedy approach handles this perfectly.

Remove Duplicates II

📋 Problem Requirement

Given a sorted array nums, remove duplicates in-place such that each unique element appears at most twice. The relative order must be preserved. Return the number of elements remaining.

Example Input

[1, 1, 1, 2, 2, 3]

Expected Output

k = 5, nums = [1, 1, 2, 2, 3]
APPROACH 1 | LINEAR SCAN & SHIFT

Intuition: "The Manual Cleanup"

Iterate through the array while keeping a count of the current number. If we encounter a third (or more) instance of the same number, we shift all elements to the right of it one position to the left. This "erases" the extra duplicate.

Ready to scan. Pointers initialized.

Java Implementation (Brute Force)

public int removeDuplicates(int[] nums) {
    int currentSize = nums.length;
    if (currentSize < 3) return currentSize;

    int currentIndex = 1;
    int sameElementCount = 1;

    while (currentIndex < currentSize) {
        if (nums[currentIndex] == nums[currentIndex - 1]) {
            sameElementCount++;
        } else {
            sameElementCount = 1;
        }

        // If we find more than 2, shift everything left
        if (sameElementCount > 2) {
            for (int k = currentIndex; k < currentSize - 1; k++) {
                nums[k] = nums[k + 1];
            }
            currentSize--;
            sameElementCount--; // We stayed at the same index, so decrement count
        } else {
            currentIndex++;
        }
    }
    return currentSize;
}

Complexity

Time: O(n²). For every extra duplicate, we shift O(n) elements.

Space: O(1). In-place modification.

Trade-off

Simple to reason about but highly inefficient for large arrays with many duplicates. Repeatedly moving data is the primary bottleneck.

APPROACH 2 | OPTIMAL TWO-POINTER

Intuition: "The Smart Overwrite"

Since the array is sorted, duplicates are grouped. We maintain a writeIndex pointer. For every element seen by the readIndex, we only write it if it's different from the element two positions back from our current write position. This automatically allows up to two duplicates.

Starting optimal scan. First two elements are always kept.

Java Implementation (Optimal)

public int removeDuplicates(int[] nums) {
    if (nums.length <= 2) return nums.length;

    // writeIndex: Position where we will write the next valid element
    int writeIndex = 2;

    // currentIndex: Explores the rest of the array
    for (int currentIndex = 2; currentIndex < nums.length; currentIndex++) {
        // Core Logic: Only keep if it's NOT a 3rd duplicate.
        // We compare against the element at (writeIndex - 2).
        if (nums[currentIndex] != nums[writeIndex - 2]) {
            nums[writeIndex] = nums[currentIndex];
            writeIndex++;
        }
    }
    
    return writeIndex;
}

Complexity

Time: O(n). Exactly one pass through the array.

Space: O(1). No auxiliary space used.

Why it works

Because the array is sorted, any value occurring more than twice will be equal to the value at writeIndex - 2. If nums[currentIndex] == nums[writeIndex - 2], it means we already have two instances of that value in our "clean" portion, so we skip the current one.

Performance Comparison

Approach Time Space Efficiency
Brute Force (Shift) O(n²) O(1) Low
Two Pointers (Optimal) O(n) O(1) Maximum

⚡ Fast Recall Cheat Sheet

The "K-Duplicates" Pattern

This problem is a specific case of "Allow at most K duplicates". The logic generalizes perfectly:

if (n > nums[write - K]) {
  nums[write++] = n;
}

Visual Memory Hook

  • 1. Write Pointer is your boundary of "clean" data.
  • 2. Always compare the New Candidate with the Backdoor (write - K).
  • 3. If they match, the current spot is "saturated". Skip it.
